machine_desc结构体

针对ARM架构，内核提供了一个重要的结构体struct machine_desc，这个结构体在内核移植中起到相当重要的作用，内核通过machine_desc结构体来控制系统体系架构相关部分的初始化，注意这个结构体只针对ARM的体系架构，在内核启动初始化过程中，进入setup_arch()阶段后的处理和相关结构体都是由各个体系架构按照自己的理解设计的，相关代码都在内核代码树的arch/$ARCH下分别管理。

machine_desc结构体定义在arch/arm/include/asm/mach/arch.h，如下所示：

struct machine_desc {
    /*
     * Note! The first four elements are used
     * by assembler code in head.S, head-common.S
     */
    unsigned int        nr;        /* architecture number    */
    unsigned int        phys_io;    /* start of physical io    */
    unsigned int        io_pg_offst;    /* byte offset for io 
                         * page tabe entry    */

    const char        *name;        /* architecture name    */
    unsigned long        boot_params;    /* tagged list        */

    unsigned int        video_start;    /* start of video RAM    */
    unsigned int        video_end;    /* end of video RAM    */

    unsigned int        reserve_lp0 :1;    /* never has lp0    */
    unsigned int        reserve_lp1 :1;    /* never has lp1    */
    unsigned int        reserve_lp2 :1;    /* never has lp2    */
    unsigned int        soft_reboot :1;    /* soft reboot        */
    void            (*fixup)(struct machine_desc *,
                     struct tag *, char **,
                     struct meminfo *);
    void            (*map_io)(void);/* IO mapping function    */
    void            (*init_irq)(void);
    struct sys_timer    *timer;        /* system tick timer    */
    void            (*init_machine)(void);
};

板级初始化

查看针对smdk2440开发板初始化的代码例子，参考arch/arm/mach-s3c2440/mach-smdk2440.c，有这么一段话：

MACHINE_START(S3C2440, "SMDK2440")
    /* Maintainer: Ben Dooks <ben@fluff.org> */
    .phys_io    = S3C2410_PA_UART,
    .io_pg_offst    = (((u32)S3C24XX_VA_UART) >> 18) & 0xfffc,
    .boot_params    = S3C2410_SDRAM_PA + 0x100,

    .init_irq    = s3c24xx_init_irq,
    .map_io        = smdk2440_map_io,
    .init_machine    = smdk2440_machine_init,
    .timer        = &s3c24xx_timer,
MACHINE_END

MACHINE_START和MACHINE_END是两个宏，同样定义在arch/arm/include/asm/mach/arch.h，如下：

/*
 * Set of macros to define architecture features.  This is built into
 * a table by the linker.
 */
#define MACHINE_START(_type,_name)            \
static const struct machine_desc __mach_desc_##_type    \
 __used                            \
 __attribute__((__section__(".arch.info.init"))) = {    \
    .nr        = MACH_TYPE_##_type,        \
    .name        = _name,

#define MACHINE_END                \
};

应用这两个宏后，上面的代码转化为：

static const struct machine_desc __mach_desc_S3C2440
 __used                            
 __attribute__((__section__(".arch.info.init"))) = {
    .nr        = MACH_TYPE_S3C2440,
    .name        = "SMDK2440",
    /* Maintainer: Ben Dooks <ben@fluff.org> */
    .phys_io    = S3C2410_PA_UART,
    .io_pg_offst    = (((u32)S3C24XX_VA_UART) >> 18) & 0xfffc,
    .boot_params    = S3C2410_SDRAM_PA + 0x100,

    .init_irq    = s3c24xx_init_irq,
    .map_io        = smdk2440_map_io,
    .init_machine    = smdk2440_machine_init,
    .timer        = &s3c24xx_timer,
};

也就是定义了一个结构体的变量叫__mach_desc_S3C2440，放在.arch.info.init节中，如果定义了多个这样的变量，则在链接内核时，它们被组织在一起，开始地址为__arch_info_begin，结束地址为__arch_info_end(这个可以从链接脚本arch/arm/kernel/vmlinux.lds.S中可以看出来)。

每一个这样的变量对应一个开发板类型。U-BOOT引导内核时，会在R1寄存器中给出开发板的标记(机器类型ID)；譬如对于SMDK2440开发板，U-Boot传入的机器类型ID为MACH_TYPE_SMDK2440。内核在启动过程中，也就是著名的__lookup_machine_type()函数过程中会从R1寄存器中读取这个值，然后到.arch.info.init节中遍历查找每个machine_desc结构体变量的.nr，同时进行比对，如果找到则完成机器(开发板)的检查，同时还根据对应的machine_desc结构体变量中保存的参数，譬如一些回调函数继续完成板一级的初始化过程。

machine_desc结构体中的几个重要的成员的设置

phys_io和io_pg_offst

参考linux下machine_desc结构体中的phys_io与io_pg_offst的作用及使用方法

从参考文章中知道自从v2.6.38中已经没有phys_io与io_pg_offs这两个变量。但我练习使用的是v2.6.32.2，所以还要涉及到它们。
字面上phys_io用于保存物理IO的起始地址，io_pg_offst用于保存IO页表的偏移字节的地址（MMU页表）。但在实际操作中phys_io用来保存UART的物理地址，io_pg_offst用来保存UART的内核空间虚拟地址。两者的映射关系在arch/arm/kernel/head.S中建立。这样，在kernel还没有初始化完MMU时，就可以通过写io_pg_offst向UART打印调试信息。这主要在low level的调试函数中使用，比如printascii()。

boot_params

这个是用来设置存放从bootloader传递到内核来的tag参数的地址信息的。在setup_arch()函数中会根据这个值到内存中找到这些参数然后调用parse_tags()函数进行解析。

注意这个参数填写的地址信息是物理地址，所以具体使用前要将其转化为虚拟地址(phys_to_virt())。

有关tag参数的概念，参考U-boot和内核之间传递复杂参数的TAG方式

void __init setup_arch(char **cmdline_p)
{
    struct tag *tags = (struct tag *)&init_tags;
    ...

    if (__atags_pointer)
        tags = phys_to_virt(__atags_pointer);
    else if (mdesc->boot_params)
        tags = phys_to_virt(mdesc->boot_params);

    ...

    if (tags->hdr.tag == ATAG_CORE) {
        ...
        parse_tags(tags);
    }

    ...
}

fixup

用内核参数列表填充meminfo，这个函数，有些板子有，有些没有，譬如我使用的mini2440就不需要。

map_io/init_irq/timer/init_machine

这几个回调函数都需要我们在系统移植的时候根据自己的板子情况自己调整。
其中：

  • map_io：处理和IO端口映射相关的事务。
  • init_irq：和中断相关
  • timer：和系统时钟相关
  • init_machine：板级设备的注册和初始化

下面来分析一下这些回调函数被调用的地方和时间点：

首先查看arch/arm/kernel/setup.c文件的setup_arch()函数中有这么几段代码：

void __init setup_arch(char **cmdline_p)
{
    ...
    struct machine_desc *mdesc;

    ...

    mdesc = setup_machine(machine_arch_type);

    ...
    paging_init(mdesc);
    ...

    /*
     * Set up various architecture-specific pointers
     */
    init_arch_irq = mdesc->init_irq;
    system_timer = mdesc->timer;
    init_machine = mdesc->init_machine;

    ...
}

其中从paging_init()函数入口一路跟踪下去会调用到map_io。参考如下流程：

/*
 * paging_init() sets up the page tables, initialises the zone memory
 * maps, and sets up the zero page, bad page and bad page tables.
 */
void __init paging_init(struct machine_desc *mdesc)
{
    ...
    devicemaps_init(mdesc);
    ...
}


static void __init devicemaps_init(struct machine_desc *mdesc)
{
    ...

    /*
     * Ask the machine support to map in the statically mapped devices.
     */
    if (mdesc->map_io)
        mdesc->map_io();

    ...
}

init_arch_irq/init_machine是分别定义的两个静态函数指针变量，而system_timer是一个静态结构体struct sys_timer的指针变量。也就是说在setup_arch()函数中根据U-boot传开的机器ID，保存在machine_arch_type中得到对应的machine_desc结构体对象的地址，然后把一些后面要使用的回调函数等等先保存在init_irq/timer/init_machine这些静态的变量中。

然后分别在后面的init_IRQ()/time_init()/customize_machine()函数中通过如下方式间接调用init_irq/timer/init_machine。

/* @ arch/arm/kernel/irq.c */
void __init init_IRQ(void)
{
    ...    

    init_arch_irq();
}

/* @ arch/arm/kernel/time.c */
void __init time_init(void)
{
    ...

    system_timer->init();
}

/* @ arch/arm/kernel/setup.c */
static int __init customize_machine(void)
{
    /* customizes platform devices, or adds new ones */
    if (init_machine)
        init_machine();
    return 0;
}
arch_initcall(customize_machine);

其中customize_machine()函数的调用路径相对稍微有点复杂：

  • 首先rest_init()中会启动内核线程，执行kernel_init()函数。
  • kernel_init()函数中调用do_basic_setup()函数;
  • do_basic_setup()函数会执行do_initcalls()函数
  • 而在do_initcalls()函数中会依次执行所有声明为__init的函数，包括customize_machine()`函数。

所以我们如果按照start_kernel()函数中第一级调用者来看，可以认为init_machine()是从rest_init()函数发起的。

所以整理一下这些函数的调用顺序：

asmlinkage void __init start_kernel(void)
{
    ...

    setup_arch(&command_line); //这里会触发map_io

    ...

    init_IRQ(); //这里会触发init_irq
    ...
    time_init(); //这里会触发timer的初始化

    ...
    /* Do the rest non-__init'ed, we're now alive */
    rest_init(); //这里会触发init_machine
}

由此可见一斑。
