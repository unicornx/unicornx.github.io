根文件系统的概念

理论上说一个嵌入式设备如果内核能够运行起来，且不需要运行用户进程的话，是不需要文件系统的。

但是内核只是运行在内存中，如果内核需要访问外存，譬如存放计算的结果，执行存放在外存上的程序，那就必须配合文件系统。文件系统的用词有好几层意思，在不同的上下文中含义不同。

在外存上下文中，文件系统常常指的是外部存储介质本身，譬如一个磁盘分区，U盘等。
在内核上下文中，文件系统指的是内核中的一个子模块，该模块的功能是负责将外存，譬如一个磁盘分区上存储的数据以一定的方式组织起来，以一棵逻辑树的形式展现给上层应用，以便上层应用访问物理存储介质上的数据同时又不必关心具体的物理介质特性。

按照内核文件系统子模块管理组织外存的方式可以将文件系统划分为很多种类型，括ext2、ext3、vfat、ntfs、iso9660、jffs、romfs和nfs等。一个外存介质采用一种文件系统方式进行管理，一个计算机系统上可能需要管理多个物理外存介质，譬如一个硬盘分区用ext3，另一个硬盘分区用ntfs，还有一个光驱用iso9660。这些文件系统都是采用挂载的方式在内核启动引导过程中和内核建立连接。

第一个挂载的文件系统称为”根文件系统“，其他的文件系统则以子树的形式挂载在根文件系统的挂载点上(目录)。

根文件系统的布局原则上都需要遵循FHSFHS(Filesystem
Hierarchy Standard)标准的建议规定。该标准规定了根目录下各个子目录的名称及其存放的内容：

目录名 存放的内容
/bin 必备的用户命令，例如ls、cp等
/sbin 必备的系统管理员命令，例如ifconfig、reboot等
/dev 设备文件，例如mtdblock0、tty1等
/etc 系统配置文件，包括启动文件，例如inittab等
/lib 必要的链接库，例如C链接库、内核模块
/home 普通用户主目录
/root root用户主目录
/usr/bin 非必备的用户程序，例如find、du等
/usr/sbin 非必备的管理员程序，例如chroot、inetd等
/usr/lib 库文件
/var 守护程序和工具程序所存放的可变，例如日志文件
/proc 用来提供内核与进程信息的虚拟文件系统，由内核自动生成目录下的内容
/sys 用来提供内核与设备信息的虚拟文件系统，由内核自动生成目录下的内容
/mnt 文件系统挂接点，用于临时安装文件系统
/tmp 临时性的文件，重启后将自动清除

制作根文件系统就是要建立以上的目录，并在其这些目录中根据内核的需要和实际项目的需要创建各种文件，建立完整目录内容。比如在/bin、/sbin目录下存放各种可执行程序，在/etc目录下存放配置文件，在/lib目录下存放库文件等等。其过程大体包括：

  • Step1: 利用busybox来生成/bin、/sbin、/usr/bin、/usr/sbin目录下的可执行文件
  • Step2: 利用交叉编译工具链中的glibc库来构建/lib目录
  • Step3: 手工构建/etc目录
  • Step4: 手工构建最简化的/dev目录
  • Step5: 配置系统自动生成/proc目录
  • Step6: 创建/sys目录并利用udev构建完整的/dev目录
  • Step7: 创建其它目录(可选)
  • Step8: 制作根文件系统的yaffs2映像文件

下面就来详细介绍根据这个过程，针对mini2440这个嵌入式系统制作一个最简单的根文件系统，不针对任何实际的项目需要，仅仅满足内核启动后能够正常运行即可。

以下步骤描述中注意以下几点：

  • 实验过程中的内核镜像直接使用了FriendlyARM提供的内核。
  • 在Step7之前测试根文件系统时都是采用的nfs的挂载方式。有关nfs环境的设置，具体参考mini2440开发环境搭建总结。
  • 给出的命令操作例子中，如果是以$作为shell的提示符开头，表示是在ubuntu主机中执行的命令；如果是以#作为shell的提示符开头，表示是在mini2440目标板系统中执行的命令。

Step1: 利用busybox来生成/bin、/sbin、/usr/bin、/usr/sbin目录下的可执行文件

这些目录下存储的主要是常用命令的二进制文件。如果要自己编写这几百个常用命令的源程序，不仅工作量巨大而且生成的文件的容量也往往很大，而我们主要针对的是嵌入式系统，对容量的大小也是有限制和考量的，好在我们有嵌入式Linux系统的瑞士军刀——busybox，事情就简单很多。

使用Busybox 可以自动生成根文件系统所需的bin、sbin、usr 目录和linuxrc文件。

1、从busybox官网下载busybox的tar包busybox-1.13.3.tar.bz2。

2、解包，tar xjvf busybox-1.13.3.tar.bz2

3、修改Makefile文件

CROSS_COMPILE   ?= arm-linux-
...
ARCH            ?= arm //这个感觉可以不要

4、执行make menuconfig对busybox进行make之前的配置。

busybox配置主要分两部分。

第一部分是Busybox Settings，主要编译和安装busybox的一些选项。这里主要需要配置：

  • Build Options -- Build BusyBox as a static binary (no shared libs)，表示编译busybox时，是否静态链接C库。我们选择动态链接C库，所以该项无需选中。

  • Installation Options -- Applets links (as soft-links) -- (X) as soft-links，表示安装busybox时，将各个命令安装为指向busybox的软链接还是硬链接7。我们选择软链接。

  • Installation Options -- BusyBox installation prefix，表示busybox的安装位置。我们可以根据自己的需要指定一个安装目录，譬如/work/nfs_root/fs_mini3。如果没有指定安装位置则会在busybox目录下生成_install`文件夹并放在这里。

  • Busybox Library Tuning。保留Command line editing以支持命令行编辑；保留History size以支持记忆历史命令；选中Tab completion和Username completion以支持命令自动补全。

第二部分是Applets，他将busybox的支持的几百个命令分门别类。我们只要在各个门类下选择想要的命令即可。这里我们基本保持默认设置，其中

  • 选中Networking Utilities -- httpd下的Enable -u <user> option，以启用http服务器的功能allows the server to run as a specific user

5、编译和安装

make all install

安装完成后，可以看到在/work/nfs_root/fs_mini3目录下生成了bin,sbin,usr/bin和usr/sbin目录，其下包含了我们常用的命令，这些命令都是指向bin/busybox的软链接，而busybox本身的大小不到800K：

Step2: 利用交叉编译工具链中的glibc库来构建/lib目录

光有应用程序（bin/busybox）是不够的，因为应用程序本身需要使用C库的库函数，由于我们在配置busy时指定的是动态链接，因此还需要制作for ARM的C库(共享库)，并将其放置于/lib目录。

交叉编译工具链下for ARM的C库是现成的，我们只需要拷贝过来就可以了。我们也不需要C库目录下的所有的文件，具体需要哪些，参考交叉工具链中的c库

我使用的是FriendlyARM提供的工具链arm-linux-gcc-4.4.3.tar.gz，解压开后，主要关注D:\ws\mini2440\rootfs\opt\FriendlyARM\toolschain\4.4.3\arm-none-linux-gnueabi\sys-root\lib下的内容。

我们拷贝的内容主要包括：

  • (1) 后缀为.so的共享库文件本身以及运行时动态链接需要的符号链接
  • (2) 动态链接库加载器ld共享库及其符号链接

第1类文件的大小也相当大。并不需要全部拷贝。需要哪些库完全取决于要运行的应用程序使用了哪些库函数。如果我们只制作最简单的系统，那么我们只需要运行busybox这一个应用程序即可。通过执行

可知：busybox只用到了2个库：通用C库（libc）、数学库（libm），因此我们只需要拷贝这2个库及其符号链接文件即可。加上第2类文件，一共6个文件。

Step3: 手工构建/etc目录

/etc目录存放的是系统程序的主配置文件，因此需要哪些配置文件取决于要运行哪些系统程序。

编写init的主配置文件inittab

即使最小的系统也一定会运行1号用户进程init，所以我们至少要手工编写init的主配置文件inittab。在这里我们有必要回顾一下内核初始化过程中涉及启动init进程的过程。这个要专门写一篇文章总结一下。总之内核会在启动的最后阶段尝试调用init进程，而在busybox环境下也就是调用busybox有关init的代码，可以参考busybox的init/init.c过程。

如果文件系统中存在/etc/inittab文件，busybox的init代码就会解析它，否则使用默认的配置。但注意我们使用的是busybox，busybox所识别的inittab文件的语法、语义与传统的SYSV的inittab有所不同。这个具体可以参考busybox源码中的examples/inittab文件，里面有详细的说明和例子，我们可以直接模仿编写自己的inittab文件。

inittab文件中每个条目用来定义一个需要init启动的子进程，并确定它的启动方式，格式为

<id>:<runlevel>:<action>:<process>。

例如：ttySAC0::askfirst:-/bin/sh

  • <id>：表示子进程要使用的控制台，若省略则使用与init进程一样的控制台
  • <runlevel>：表示运行级别，busybox init程序这个字段没有意义
  • <action>：表示init进程如何控制这个子进程，可以取以下值：
|action | 含义 |
|-------|------|
|sysinit   |系统启动后最先执行，只执行一次，init进程等待它结束后才继续执行其它动作|
|wait      |系统执行完sysinit条目后执行，只执行一次，init进程等待它结束后才继续执行其它动作|
|once      |系统执行完wait条目后执行，只执行一次，init进程不等待它结束|
|respawn   |启动完once进程后，init进程监测发现子进程退出时，重新启动它|
|askfirst  |启动完respawn进程后，与respawn类似，不过init进程先输出” Please press Enter to activate this console“，等用户输入回车后才启动子进程|
|shutdown  |当系统重启，关机时|
|restart   |Busybox中配置了CONFIG_FEATURE_USE_INITAB，并且init进程接收到SIGUP信号时执行，先重新读取、解析/etc/inittab文件，再执行restart程序|
|ctrlaltdel|按下ctrl+alt+del键时执行，不过在串口控制台中无法输入它|
  • <process>：表示进程对应的二进制文件。如果前面有-号，表示该程序是“可以与用户进行交互的”

我们制作最简单的/etc/inittab文件，其内容如下：

::sysinit:/etc/init.d/rcS
::askfirst:-/bin/sh
::ctrlaltdel:/sbin/reboot
::shutdown:/bin/umount -a –r

第一行表示init进程启动的第一个子进程是/etc/init.d/rcS，它是一个脚本，可以在里面指定用户想要执行的操作
第二行表示在sysinit完成之后会启动shell，由于没有指定则使用与init进程一样的控制台，即/dev/console，也可以指定自己的控制台。(TBD，控制台的概念有待理清)
第三行表示按下ctrl+alt+del键时执行/sbin/reboot，不过在串口控制台中无法输入它
第四行表示当系统重启，关机时执行umount操作。

制作最简单的脚本程序文件/etc/init.d/rcS

前面在编写/etc/inittab文件时已经看到，我们已经指定init进程启动的第一个子进程是/etc/init.d/rcS，它是一个脚本，可以在里面指定用户想要执行的操作，譬如配置IP地址，挂载其他的文件系统等。
目前为简化操作，可以先设置为空，其内容如下，等我们后面再慢慢完善它：

#!/bin/sh

修改shell脚本文件/etc/init.d/rcS的权限，以使其可被执行：

$ chmod a+x /etc/init.d/rcS

Step4: 手工构建最简化的/dev目录

先创建/dev目录

$ mkdir /dev

我们先手动创建一些必需的设备文件，之所以称为必需是因为没有这些文件，内核启动过程中会报错。这样我们可以先快速测试一下这个最简单的根文件系统。

创建/dev/console设备文件

前面我们也看到了，参考内核init/main.c文件的init_post()函数。该函数会在启动init进程之前先尝试打开/dev/console文件。如果不存在这个文件会导致启动Linux操作系统时显示：

VFS: Mounted root (nfs filesystem).
Freeing init memory: 112K
Warning: unable to open an initial console.

创建方法:

$ sudo mknod console c 5 1

创建/dev/null设备文件

参考busybox的init/init.c，如果内核提供的控制台打不开，就会尝试打开/dev/null，如果不存在这个文件会导致启动Linux操作系统时显示：

VFS: Mounted root (nfs filesystem).
Freeing init memory: 112K
init: can't open '/dev/null': No such file or directory

创建方法:

$ sudo mknod null c 1 3

此时再次重启开发板上的linux，应该已经可以看到系统启动成功进入控制台。

VFS: Mounted root (nfs filesystem).

starting pid 229, tty '': '/etc/init.d/rcS'

Please press Enter to activate this console.
starting pid 231, tty '': '/bin/sh'
#

注意: 其他的设备文件可以利用Step6介绍的方法在/dev下自动创建。

Step5: 配置系统自动生成/proc目录

proc文件系统是一个重要的用来提供内核与进程信息的虚拟文件系统，它没有实际的存储设备，里面的内容，包括子目录和文件都是由内核自动在访问的时候临时生成的，用来读取系统的运行状态，也可以设置一些文件的内容来控制系统的行为。很多命令都是通过解析/proc目录里的内容来工作的，比如最常用的ps命令，如果该目录不存在或者内容为空的话，这些命令都会工作不正常。

这个文件系统一般挂载在/proc目录下。所以我们要做的事情就是在根目录下创建一个名字叫做proc的目录作为该虚拟文件系统的挂载点，然后执行如下命令挂载该文件系统：

mount -t proc none /proc

对于mount -t proc none /proc的解释：通常情况下mount命令应该写为mount –t ext2 /dev/hdb1 /proc。但由于现在挂载的/proc是虚拟文件系统，它不与任何物理硬盘分区相对应，因此在表示物理硬盘分区的位置用占位符none来表示。

更方便，也是标准的做法是将该文件系统在init进程的sysinit阶段完成该文件系统的挂载：

具体操作如下：

  1. 创建/proc目录

    $ mkdir /proc

  2. 修改/etc/init.d/rcS文件，添加一句话mount -a如下:

    #!/bin/sh
mount -a

    mount -a命令会读取/etc/fstab文件的内容，根据该文件指定挂载的文件系统实现在启动初期的自动挂载。所以为此我们要实现一个配套的/etc/fstab文件。具体有关mount命令和/etc/fstab的信息，参考man手册的mount(8)和fstab(5)。

  3. 创建/etc/fstab文件，可以从已有系统，如Ubuntu上拷贝一个做为模版，清除已有的内容，添加一行如下：

    proc            /proc           proc    defaults        0       0

Step6: 创建/sys目录并利用udev构建完整的/dev目录

在第4步中我们在/dev目录下手动地创建了两个设备文件，但是仅有这两个文件是远远不够的，在Linux系统中每一个用户可以访问的设备都必须在/dev目录下有一个名字，否则无法访问。在任何一台linux机器上，执行ls /dev都可以看到几百个设备文件。那么我们如何为所有的需要访问的物理设备创建设备文件呢？方法一共有三种：

  • 第一种是手动创建静态设备文件：就和我们在Step4中做得一样，需要使用mknod命令事先创建很多设备文件，麻烦不说，而且是静态创建，无法支持热插拔。
  • 第二种是使用devfs：但由于在使用上存在一些问题，在2.6.13后已被废弃。
  • 第三种就是使用udev(user dev)，这也是我们在Step6中将采用的方法。

udev的工作原理是：内核基于一个特殊的虚拟文件系统sysfs，该文件系统一般挂载在/sys目录下，讷河在启动的时候将识别到的所有设备的信息自动导出到/sys目录，然后用户态的应用程序udev根据/sys中的设备信息，按照一定的规则自动在/dev目录下创建对应的设备文件。

在busybox的环境下，采用的是一个udev的简化版本mdev。具体mdev的用法可以参考busybox源码下的docs/mdev.txt文件。

在Step5操作完成之后启动的开发板系统shell下先做一些简单的测试尝试如下操作：

# mkdir /sys
# mount -t sysfs none /sys
# ls /sys

bus       devices   fs        module
# ls /dev
console  null
# mdev -s
# ls -l dev | wc -l
140

sd 0:0:0:0: [sda] 12560384 512-byte hardware sectors (6431 MB)
sd 0:0:0:0: [sda] Write Protect is off
sd 0:0:0:0: [sda] Assuming drive cache: write through
sda: sda1
sd 0:0:0:0: [sda] Attached SCSI removable disk

运行如下命令，可以看到mdev自动识别到新设备并且帮助我们在/dev目录下创建了对应文件。

# ls /dev/sda*
/dev/sda   /dev/sda1

可是，当我们将U盘拔出后，发现/dev/sda1并不自动消失；手工删除/dev/sda1后，再重新插入U盘，/dev/sda1也不会自动生成。我们需进行如下操作，以使系统能够实现即插即用：

# echo /sbin/mdev > /proc/sys/kernel/hotplug

这样插拔U盘就工作正常了。

当然我们需要将以上操作在系统启动过程中自动化完成。具体的正式步骤如下，注意这里我将Step4和Step6合并起来描述我们应该如何在根文件系统中创建/dev目录及其内容，所以假设当前还不存在/dev目录。

  1. 创建/dev目录

    $ mkdir /dev

  2. 手动创建/dev/console和/dev/null这两个文件。

    这两个文件必须要手动创建，即使我们在后继步骤中会将/dev目录挂载为tmpfs文件系统，也必须要保证这两个文件，原因就在于还在内核开始挂载根文件系统之前(自然也在挂在其他文件系统之前)的启动过程中就要求着两个文件存在，否则启动会失败并报错。

    $ cd /dev
$ sudo mknod console c 5 1
$ sudo mknod null c 1 3

  3. 创建/sys目录，即为sysfs文件系统准备好挂载点。

    $ mkdir /sys

  4. 修改/etc/fstab文件，支持自动挂载sysfs文件系统

    添加如下一行在/etc/fstab文件里：

    sysfs           /sys            sysfs   defaults        0       0

  5. 修改/etc/init.d/rcS文件，启动mdev

    添加如下两行在/etc/init.d/rcS文件里：

    echo /sbin/mdev > /proc/sys/kernel/hotplug
mdev -s

    重启系统，测试一下，上电后会发现目标板启动正常，并在/sys和/dev下都出现了很多文件。

    插拔U盘，会看到相应的设备文件被自动创建和删除。

    关电后查看Ubuntu主机上nfs目录下的根文件系统，在/dev下同样出现了很多文件，但/sys和/proc下依然空空如也，道理很简单，/sys和/proc是虚拟文件系统。

  6. 使用tmpfs挂载/dev目录

    做完上面的测试，似乎我们的根文件系统已经相当完善了。但仔细想一想，对于/dev下的文件，只要有设备插拔发生，就会在该目录下创建删除文件，即会刷新实际的存储设备nandflash芯片，再想想Nand flash的擦写寿命是有限的这个事实。似乎担心起存储设备的使用寿命的问题来。

    解决这个问题的方法就是使用tmpfs文件系统来存放/dev目录下的内容。

    有关tmpfs的介绍，参考以下内容：
    Linux的tmpfs文件系统
    Linux下的tmpfs文件系统

    结合参考文献中提到的tmpfs这个文件系统的作用，我们就应该明白，我们应该将一些内容经常会更新，而且只在上电运行阶段才关心的目录，譬如这里的/dev目录挂载为tmpfs文件系统就好了。

    修改/etc/fstab文件，添加如下一行在/etc/fstab文件里：

    tmpfs           /dev            tmpfs   defaults        0       0

    注意，修改完后如果要测试效果，并且如果已经执行过Step6中的以上步骤，记得将Ubuntu主机上的/dev目录清空一下同时保证/dev/console和/dev/null两个文件还是要手动先创建好了的。

    重启系统，测试一下，上电后会发现目标板启动正常，并在/sys和/dev下都出现了很多文件。

    但是关电后查看Ubuntu主机上nfs目录下的根文件系统，此时/dev下就只剩下两个console和null文件，其他的设备文件都不见了。

Step7: 创建其它目录(可选)

作为一个最简单的测试用根文件系统，这步不一定要做。

注意如果要建一些譬如/tmp和/var目录，依然可以挂载为tmpfs文件系统。

Step8: 制作根文件系统的yaffs2映像文件

根文件系统已经制作完毕，但目前开发板只能将其作为网络根文件系统通过nfs挂载启动，如果要烧入开发板，最后一个步骤是将其打包生成一个映像文件，以供bootloader将其烧录到nand flash上。
