理解u-boot的Makefile(基于v2009.11)

发表于 2015-06-28   |   分类于 Bootloader   |

最新更新时间：2015-07-17

参考：
U-boot版本与移植概况

本文主要分析了uboot的Makefile的执行步骤，这里只分析主要的步骤，中间如果有些小的操作就忽略了。
注：我目前研究的是uboot版本是2009.11，参考的是ARM的versatile pb开发板

设置版本

VERSION = 2009
PATCHLEVEL = 11
SUBLEVEL =
EXTRAVERSION =
ifneq "$(SUBLEVEL)" ""
U_BOOT_VERSION = $(VERSION).$(PATCHLEVEL).$(SUBLEVEL)$(EXTRAVERSION)
else
U_BOOT_VERSION = $(VERSION).$(PATCHLEVEL)$(EXTRAVERSION)
endif
TIMESTAMP_FILE = $(obj)include/timestamp_autogenerated.h
VERSION_FILE = $(obj)include/version_autogenerated.h

这里都是些有关版本号的配置，参考README的“Versioning:”章节，u-boot对其版本的定义方式如下：

Versioning:
===========

U-Boot uses a 3 level version number containing a version, a
sub-version, and a patchlevel: "U-Boot-2.34.5" means version "2",
sub-version "34", and patchlevel "4".

The patchlevel is used to indicate certain stages of development
between released versions, i. e. officially released versions of
U-Boot will always have a patchlevel of "0".

就是说u-boot的版本定义方式为3层，..。但对应的实际是Makefile文件中的VERSION，PATCHLEVEL和SUBLEVEL，感觉文档更新有点慢。

定义 对应Makefile变量 老版本格式 新版本格式
version VERSION 1 2008
sub-version PATCHLEVEL 3 10
patch-level SUBLEVEL 4
extra-info EXTRAVERSION -rc1
version-string U_BOOT_VERSION 1.3.4 2008.10-rc1

设置主机host的编译环境

因为是嵌入式开发，采用的是交叉编译，所以需要设置主机的编译环境，包括：获取主机的arch和os；设置shell。

HOSTARCH := $(shell uname -m | \
    sed -e s/i.86/i386/ \
        -e s/sun4u/sparc64/ \
        -e s/arm.*/arm/ \
        -e s/sa110/arm/ \
        -e s/powerpc/ppc/ \
        -e s/ppc64/ppc/ \
        -e s/macppc/ppc/)

HOSTOS := $(shell uname -s | tr '[:upper:]' '[:lower:]' | \
        sed -e 's/\(cygwin\).*/cygwin/')

# Set shell to bash if possible, otherwise fall back to sh
SHELL := $(shell if [ -x "$$BASH" ]; then echo $$BASH; \
    else if [ -x /bin/bash ]; then echo /bin/bash; \
    else echo sh; fi; fi)

export    HOSTARCH HOSTOS SHELL

# Deal with colliding definitions from tcsh etc.
VENDOR=

设置编译输出的存放路径

检查是否禁止在编译过程中向终端输出信息，利用findstring函数检查变量 MAKEFLAGS 是否为参数’s’
当编译U-boot的时候，我们会输入make [选项] [目标] …如果输入:make -s，则MAKEFLAGS = s
XECHO = echo 编译时将输出编译过程的信息，否则安静编译,不输出信息

#########################################################################
# Allow for silent builds
ifeq (,$(findstring s,$(MAKEFLAGS)))
XECHO = echo
else
XECHO = :
endif

uboot允许将生成的所有编译结果都存放在源代码主目录以外的一个单独的目录下，这样就可以保持源代码目录比较干净，这个目录用变量BUILD_DIR来表示，缺省为空，即为uboot源代码目录的主目录。这个变量可以修改，具体的设置方法，可以参考以下注释。一般不需要指定，就在当前源代码主目录下生成，所以直接“make all”就好。 

#########################################################################
#
# U-boot build supports producing a object files to the separate external
# directory. Two use cases are supported:
#
# 1) Add O= to the make command line
# 'make O=/tmp/build all'
#
# 2) Set environement variable BUILD_DIR to point to the desired location
# 'export BUILD_DIR=/tmp/build'
# 'make'
#
# The second approach can also be used with a MAKEALL script
# 'export BUILD_DIR=/tmp/build'
# './MAKEALL'
#
# Command line 'O=' setting overrides BUILD_DIR environent variable.
#
# When none of the above methods is used the local build is performed and
# the object files are placed in the source directory.
#

ifdef O
ifeq ("$(origin O)", "command line")
BUILD_DIR := $(O)
endif
endif

ifneq ($(BUILD_DIR),)
saved-output := $(BUILD_DIR)

# Attempt to create a output directory.
$(shell [ -d ${BUILD_DIR} ] || mkdir -p ${BUILD_DIR})

# Verify if it was successful.
BUILD_DIR := $(shell cd $(BUILD_DIR) && /bin/pwd)
$(if $(BUILD_DIR),,$(error output directory "$(saved-output)" does not exist))
endif # ifneq ($(BUILD_DIR),)

注意这里定义的几个变量：

  • OBJTREE： 如果指定了编译目录BUILD_DIR的值，就采用$(BUILD_DIR)替换它，否则就等于$(CURDIR)，CURDUR是Make自定义的变量，指当前工作路径，因为我们在编译make之前都要cd到uboot源代码的主目录下，所以其值就等于uboot源代码的主目录的文件系统路径。
  • SRCTREE：设置为$(CURDIR)
OBJTREE        := $(if $(BUILD_DIR),$(BUILD_DIR),$(CURDIR))
SRCTREE        := $(CURDIR)
TOPDIR        := $(SRCTREE)
LNDIR        := $(OBJTREE)
export    TOPDIR SRCTREE OBJTREE

MKCONFIG    := $(SRCTREE)/mkconfig
export MKCONFIG

ifneq ($(OBJTREE),$(SRCTREE))
REMOTE_BUILD    := 1
export REMOTE_BUILD
endif

仔细看看这段注释，obj和src也是两个编译中重要的变量，会影响后面编译的输出路径。如果我们指定了BUILD_DIR，则$(OBJTREE)和$(SRCTREE)的值会不同，那么obj和src会相应地被赋予$(OBJTREE)和$(SRCTREE)的值，否则就都是为空，即缺省为当前主目录。

# $(obj) and (src) are defined in config.mk but here in main Makefile
# we also need them before config.mk is included which is the case for
# some targets like unconfig, clean, clobber, distclean, etc.
ifneq ($(OBJTREE),$(SRCTREE))
obj := $(OBJTREE)/
src := $(SRCTREE)/
else
obj :=
src :=
endif
export obj src

# Make sure CDPATH settings don't interfere
unexport CDPATH

Makefile文件从这里开始往下就是make的规则描述了，一个Makefile一般会包括很多规则，U-boot的规则整体上由三部分组成：

  • 生成u-boot镜像的终极规则及其所依赖的规则

  • 生成针对各种开发板的配置开关项列表的规则

  • 其他辅助规则

下面我们依次分别来看一下。

生成u-boot最终img的主要规则

#########################################################################

ifeq ($(ARCH),powerpc)
ARCH = ppc
endif

# The "tools" are needed early, so put this first
# Don't include stuff already done in $(LIBS)
SUBDIRS    = tools \
      examples/standalone \
      examples/api

.PHONY : $(SUBDIRS)

上面还是一些准备工作，SUBDIRS这个变量会保存编译时需要访问的所有子目录的名称，用空格分隔，make时会按照SUBDIRS中子目录出现的顺序，依次进入并制作。根据注释，因为make时会需要用到一些uboot自制的tools，所以将tools目录列为SUBDIRS的第一个成员，然后依次是examples下的一些东东，后面还会对这个SUBDIRS变量追加子目录。

一个重要的判断

ifeq ($(obj)include/config.mk,$(wildcard $(obj)include/config.mk))

上面这句ifeq判断的话很重要，首先这句话是一个if判断，在判断什么呢？wildcard的格式是“wildcard PATTERN”，PATTERN表达式一般是个文件路径，wildcard会根据PATTERN表达式匹配并查找该文件是否存在，如果存在就将其路径字符串返回，否则就返回空，所以可以用wildcard来查找一个文件是否存在。这里就是在判断“$(obj)include/config.mk”这个文件是否存在。假设我们没有指定编译输出目录，则$(obj为空，所以就是在检查“include/config.mk”这个文件是否存在，这个文件是我们在编译之前执行形如make xxx_config的命令创建出来的，所以通过判断该文件是否存在可以知道我们在make之前是否执行了make xxx_config。有关make xxx_config干了些什么，参考理解u-boot的make xxx_config机制(基于v2009.11)

注意这个ifeq其对应的endif在下面很远的地方，具体是在Makefile文件的第488行，而且还是个二路分支判断语句，去掉内部的大段逻辑，总框架如下：

ifeq ($(obj)include/config.mk,$(wildcard $(obj)include/config.mk)) # 第148行
# 分支一，可编译最终结果相关的一系列规则描述，此处跳过N行
... 
else    # !config.mk 第481行
# 分支二： 
all $(obj)u-boot.hex $(obj)u-boot.srec $(obj)u-boot.bin \
    $(obj)u-boot.img $(obj)u-boot.dis $(obj)u-boot \
    $(SUBDIRS) $(TIMESTAMP_FILE) $(VERSION_FILE) gdbtools updater env depend \
dep tags ctags etags cscope $(obj)System.map:
    @echo "System not configured - see README" >&2
    @ exit 1， 
endif    # config.mk 第488行

所以这个判断的作用就是确保在make或者make all之前我们必须针对你需要移植的开发板运行过make xxx_config。判断的原则就是看$(obj)include/config.mk这个文件是否存在，如果存在说明运行过配置，就允许继续make，否则会打印错误告警如下：

fs@ubuntu$ make
System not configured - see README
make: *** [all] Error 1

所以我们重点看分支一，走进分支一就是在具体执行制作u-boot二进制镜像的编译工作了，主要规则都在这里了。

导入(include)重要的配置文件(.mk)

导入$(obj)include/autoconf.mk

# Include autoconf.mk before config.mk so that the config options are available
# to all top level build files.  We need the dummy all: target to prevent the
# dependency target in autoconf.mk.dep from being the default.
all:

第153行出现的all是个假的终极目标，我们输入make all时并不是运行它，因为这个all既没有依赖，也没有规则，它下面的sinclude语句不是它的动作，原因很简单，因为前面没有TAB。这个all的用途在它前面的注释里说得很清楚了。

sinclude $(obj)include/autoconf.mk.dep
sinclude $(obj)include/autoconf.mk

所以make继续从154行开始执行“sinclude”，参考Makefile中include、-include、sinclude的区别,总结如下：

  • 使用“include FILENAMES…”，make程序处理时，如果“FILENAMES”列表中的任何一个文件不能正常读取而且不存在一个创建此文件的规则时make程序将会提示错误并退出。
  • 使用“-include FILENAMES…”的情况是，当所包含的文件不存在或者不存在一个规则去创建它，make程序会继续执行，只有真正由于不能正确完成终极目标的重建时（某些必需的目标无法在当前已读取的makefile文件内容中找到正确的重建规则），才会提示致命错误并退出。
  • 为了和其它的make程序进行兼容。也可以使用“sinclude”来代替“-include”（GNU所支持的方式）。

autoconf.mk和autoconf.mk.dep这些文件是在后继规则中被创建出来的，具体的位置要看Makefile的第465行。如果是第一次干净的build，还没有被创建，所以sinclude的意思就是如果这些文件还不存在就忽略这两条命令，继续。

导入$(obj)include/config.mk

# load ARCH, BOARD, and CPU configuration
include $(obj)include/config.mk
export    ARCH CPU BOARD VENDOR SOC

这时候$(obj)include/config.mk被导入，这个文件是通过运行make xxx_config产生的。这一步必须成功也一定会成功，因为根据前述ifeq的逻辑，逻辑走到这里说明已经做过make xxx_config。

注意这里执行的导出ARCH，CPU，BOARD，VENDOR，SOC的行为。这些记录在$(obj)include/config.mk中的变量的值会直接导致后继导入的$(TOPDIR)/config.mk和生成autoconf.mk文件时代码中所有引用这些变量的地方执行字符串替换，从而使相关的配置只针对make第一阶段，即make xxx_config针对的特定开发板。

# set default to nothing for native builds
ifeq ($(HOSTARCH),$(ARCH))
CROSS_COMPILE ?=
endif

当执行交叉编译时，需要在执行make命令时指定CROSS_COMPILE，有如下两种方法：

  • 方法一，在调用make命令时直接指定，譬如：

    $ make all CROSS_COMPILE=arm-linux-gnueabi-

  • 方法二：调用make命令前先定义该变量，再make，譬如：

    $ export CROSS_COMPILE=arm-linux-gnueabi-
$ make all

如果不指定就认为是本地编译。

导入$(TOPDIR)/config.mk

# load other configuration
include $(TOPDIR)/config.mk

注：这里include是uboot顶层目录下的config.mk文件。这也是一个重要的make配置文件，参考理解u-boot的顶层config.mk(基于v2009.11)。

终极规则(all)及其主要依赖规则

走到这里，终于看到了我们的终极目标定义all。 

#########################################################################
#########################################################################

# Always append ALL so that arch config.mk's can add custom ones
ALL += $(obj)u-boot.srec $(obj)u-boot.bin $(obj)System.map $(U_BOOT_NAND) $(U_BOOT_ONENAND)

all:        $(ALL)

这里我们看到了u-boot这个项目最终希望制作出来的都是些什么东东，即运行make all的结果：

all依赖于u-boot.srec，u-boot.bin，System.map，而u-boot.srec，u-boot.bin，System.map都是由u-boot转化而来，所以它们都依赖于u-boot。

文件名称 解释
u-boot ELF格式的U-boot文件
u-boot.srec Motorola S-Record格式的U-boot文件
u-boot.bin 最终烧写到开发板的二进制可执行的U-boot文件
System.map U-Boot文件的符号表 
$(obj)u-boot.hex:    $(obj)u-boot
        $(OBJCOPY) ${OBJCFLAGS} -O ihex $< $@

$(obj)u-boot.srec:    $(obj)u-boot
        $(OBJCOPY) -O srec $< $@

$(obj)u-boot.bin:    $(obj)u-boot
        $(OBJCOPY) ${OBJCFLAGS} -O binary $< $@

$(obj)u-boot.ldr:    $(obj)u-boot
        $(obj)tools/envcrc --binary > $(obj)env-ldr.o
        $(LDR) -T $(CONFIG_BFIN_CPU) -c $@ $< $(LDR_FLAGS)

$(obj)u-boot.ldr.hex:    $(obj)u-boot.ldr
        $(OBJCOPY) ${OBJCFLAGS} -O ihex $< $@ -I binary

$(obj)u-boot.ldr.srec:    $(obj)u-boot.ldr
        $(OBJCOPY) ${OBJCFLAGS} -O srec $< $@ -I binary

$(obj)u-boot.img:    $(obj)u-boot.bin
        ./tools/mkimage -A $(ARCH) -T firmware -C none \
        -a $(TEXT_BASE) -e 0 \
        -n $(shell sed -n -e 's/.*U_BOOT_VERSION//p' $(VERSION_FILE) | \
            sed -e 's/"[     ]*$$/ for $(BOARD) board"/') \
        -d $< $@

$(obj)u-boot.kwb:       $(obj)u-boot.bin
        $(obj)tools/mkimage -n $(KWD_CONFIG) -T kwbimage \
        -a $(TEXT_BASE) -e $(TEXT_BASE) -d $< $@

$(obj)u-boot.sha1:    $(obj)u-boot.bin
        $(obj)tools/ubsha1 $(obj)u-boot.bin

$(obj)u-boot.dis:    $(obj)u-boot
        $(OBJDUMP) -d $< > $@

GEN_UBOOT = \
        UNDEF_SYM=`$(OBJDUMP) -x $(LIBBOARD) $(LIBS) | \
        sed  -n -e 's/.*\($(SYM_PREFIX)__u_boot_cmd_.*\)/-u\1/p'|sort|uniq`;\
        cd $(LNDIR) && $(LD) $(LDFLAGS) $$UNDEF_SYM $(__OBJS) \
            --start-group $(__LIBS) --end-group $(PLATFORM_LIBS) \
            -Map u-boot.map -o u-boot
$(obj)u-boot:    depend $(SUBDIRS) $(OBJS) $(LIBBOARD) $(LIBS) $(LDSCRIPT) $(obj)u-boot.lds
        $(GEN_UBOOT)

u-boot依赖于以下目标

依赖的目标 制作该目标的动作
depend 对于$(SUBDIRS)，cpu/$(CPU)，$(dir $(LDSCRIPT))中的每个元素都进入该目录执行“make _depend”，生成各个子目录的.depend文件，.depend列出每个目标文件的依赖文件。
仅仅有目标和依赖，没有命令的，所以调用该目标时是通过隐含规则执行隐含命令的。
注意depend目标仅仅是生成了文件.depend，但是并没有执行生成任何的目标文件（比如start.o）二进制文本的任何命令，也就是说在顶层目录make depend后，在cpu/$(CPU)目录下并没有start.o文件
$(SUBDIRS) 执行tools ，examples/standalone ，examples/api目录下的Makefile
$(OBJS) 对于OBJS包含的每个成员，都进入cpu/$(CPU)目录（即cpu/arm920t）编译它们。
$(LIBBOARD) 查看对应规则
$(LIBS) 对于LIBS中的每个成员，都进入相应的子目录执行“make”命令编译它们。例如对于LIBS中的“common/libcommon.a”成员，程序将进入common目录执行Makefile，生成libcommon.a
$(LDSCRIPT) 注意对于versatile开发板，$(LDSCRIPT)是定于在lib_arm/config.mk里，LDSCRIPT := $(SRCTREE)/cpu/$(CPU)/u-boot.lds所以是cd到cpu/arm926ejs/目录下，执行make u-boot.lds命令
$(obj)u-boot.lds 将cpu/arm926ejs/u-boot.lds经cpp预处理后输出到U-Boot顶层目录下的u-boot.lds文件。

生成autoconf.mk的规则

autoconf.mk文件是一个很重要的有关配置的文件。这个文件是在make过程中自动生成的。打开autoconf.mk文件看看，就会看到形如下的内容：

CONFIG_SYS_GBL_DATA_SIZE=128
CONFIG_SYS_LONGHELP=y
CONFIG_SYS_LOAD_ADDR="0x7fc0"
CONFIG_STACKSIZE="(128 * 1024)"
CONFIG_BOOTDELAY=2
CONFIG_SYS_HELP_CMD_WIDTH=8
CONFIG_SYS_TIMER_CTRL="0x84"
...

这些以CONFIG_开头的宏就是一系列的配置开关，会影响后继的编译过程。

根据Makefile文件下面的注释，autoconfig.mk文件会被所有的Makefile文件包含，具体说是因为所有子目录的Makefile都有这么一句话include (TOPDIR)/config.mk，即包含了顶层目录下的config.mk文件，而顶层目录下的config.mk文件里有这么一行sinclude $(OBJTREE)/include/autoconf.mk。。

autoconfig.mk.dep文件的内容是autoconfig.mk文件产生时所依赖的文件的列表，这个列表中的任意一个头文件发生修改变化都会导致automake.mk文件重新生成一次，其内容如下所示：

include/autoconf.mk: include/common.h \
 /home/fs/temp/u-boot-2009.11/include/config.h \
 /home/fs/temp/u-boot-2009.11/include/configs/versatile.h \
 /home/fs/temp/u-boot-2009.11/include/asm/config.h \
 /home/fs/temp/u-boot-2009.11/include/linux/bitops.h \
 /home/fs/temp/u-boot-2009.11/include/asm/types.h \

至于autoconfig.mk和autoconfig.mk.dep里面的内容是哪里来的，让我们来仔细看一下下面这两条规则：

#
# Auto-generate the autoconf.mk file (which is included by all makefiles)
#
# This target actually generates 2 files; autoconf.mk and autoconf.mk.dep.
# the dep file is only include in this top level makefile to determine when
# to regenerate the autoconf.mk file.
$(obj)include/autoconf.mk.dep: $(obj)include/config.h include/common.h
    @$(XECHO) Generating $@ ; \
    set -e ; \
    : Generate the dependancies ; \
    $(CC) -x c -DDO_DEPS_ONLY -M $(HOSTCFLAGS) $(CPPFLAGS) \
        -MQ $(obj)include/autoconf.mk include/common.h > $@

$(obj)include/autoconf.mk: $(obj)include/config.h
    @$(XECHO) Generating $@ ; \
    set -e ; \
    : Extract the config macros ; \
    $(CPP) $(CFLAGS) -DDO_DEPS_ONLY -dM include/common.h | \
        sed -n -f tools/scripts/define2mk.sed > $@.tmp && \
    mv $@.tmp $@

在自动make过程中，这两条规则会先执行第二条规则，运行预处理命令，选项”-dM”的作用是提取include/common.h中定义的宏，然后输出给tools/scripts/define2mk.sed脚本处理，提取所有的config marcos，即配置宏，保存到autoconf.mk文件中。

至于autoconf.mk的内容，因为include/common.h该文件又包含了$(obj)include/config.h，$(obj)include/config.h这个文件是我们执行make xxx_config生成的。以我操作的versatile开发板为例，执行make versatile_config后生成的’include/config.h’里面的内容如下：

#define CONFIG_ARCH_VERSATILE_PB

/* Automatically generated - do not edit */
#include <configs/versatile.h>
#include <asm/config.h>

所以autoconf.mk文件中出现的所有”CONFIG_”宏的来源就是来自于include/common.h，include/config.h，include/configs/versatile.h以及include/asm-arm/config.h等等一系列的头文件，由于头文件会嵌套包含，所以autoconf.mk里的这些宏究竟依赖与多少头文件，这就是列在autoconfig.mk.dep里的内容了。
