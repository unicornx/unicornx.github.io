一、NAND flash的存储单元组织结构

NAND-Flash上存放的数据的最小单位是bit，我们可以想象每个bit只能表示0或者1两种状态。每个bit保存在独立的存储单元(memory cell)中，一般来说，一个cell中只能存储一个bit。这些cell以8个或者16个为一组，连成bit-line，即所谓的byte(x8)/word(x16)，这就是所谓NAND-Flash芯片的位宽。多个bit-line组成一个page，多个page会再形成一个block，多个block最终构成一片NAND-Flash芯片。

page是NAND-Flash进行”读”或者”写”的基本单位，之所以引入block的概念是因为在进行”写”入操作之前必须对NAND-flash先执行所谓的”擦除”动作，而”擦除”则以block为单位进行的。
对于每一个page中的所有bit-line分为main区和spare区，main区用于正常数据的存储，之所以在存放正常数据的main区之外还存在spare区正是因为NAN-flash相对于NOR-flash有一个劣势是很容易产生坏块，因此在使用NAND-Flash时，往往在初始化/格式化的时候要利用校验算法发现坏块并标注出来，以便以后不再使用该坏块，标注的结果就是存放在spare区中，所以spare区中存放的信息其中很重要的一项就是标记该page所从属的block是好的还是坏的，除此之外，spare区还用于存储一些附加信息，譬如该page所从属的block的逻辑地址、页内数据的ECC校验和等。

以我手头的mini2440所搭配的NAND-Flash芯片K9F2G08U0B为例，单片容量包括2048个block，每个block包含64个page，每片page上共有2112个bit-line，其中2048个bit-line构成其main区，剩下的64个bit-line构成其spare区。由于K9F2G08U0B其位宽为8，即一个bit-line包含8个bit，正好一个字节Byte，而我们平时讨论存储的基本单位就是Byte，所以K9F2G08U0B的每片page上的容量就是2112字节，其中2048(2K)字节存放正常数据，64字节存放附加信息。
综上所述，该芯片的总容量是2048*64*2112字节=264M字节，但其中存放有效数据的总容量为2048*64*2048字节=256M字节，其余的2048*64*64字节=8M字节为spare区域。

地址传输：

比较复杂。由前述，以K9F2G08U0B为例，该芯片以8位(字节)为单位寻址，需要地址线为29根。具体为何是29根，根据前述NAND flash的存储单元组织结构，如果我们把flash上的字节铺开看成是线性的，则每个字节的地址可以看成由以下三段组成：

Block_Address * Page_Address * Column_Address

其中：

  • Block_Address是芯片内block的编号，以K9F2G08U0B为例就是总共2048个block，编号0~2047，需要11位进行地址编码
  • Page_Address是每个block内部page的编号，以K9F2G08U0B为例每个block包含64个page，编号0~63，需要6位进行地址编码
  • Column_Address是对应每个page内部bit-line的编号，以K9F2G08U0B为例每个page包含2112个bit-line，需要至少12位进行地址编码。

所以总共加起来，需要11+6+12=29位。

还有一种简单的估算方法是以K9F2G08U0B为例其总容量是264M字节，256M需要至少28根，512M需要至少29根，264M介于256M和512M之间所以至少29根线才可以访问，用不完但也不能比29根少。

其次特别地在传输地址时，如前所述需要将很宽的地址分解为多个批次，每批次不超过8位，分批传送给芯片。以K9F2G08U0A为例，参考下图，因为高达29位，所以分成5个周期来实现。

其中第一个周期和第二个周期传输A0~A11，共12位，手册上称为”Column Address”，实际对应前述的”Column_Address”的概念，即page内的bit-lines的编号。
剩下的三个周期，第三个周期访问的地址为A12~A19；第四个周期访问的地址为A20~A27；第五个周期访问的地址为A28，这三个周期共寻址17位，在手册上统称为”Row Address”，但实际上包含了前述的”Page_Address”(A12~A17)和”Block_Address”(A18~A28)。
注，A0~A17在进行擦除操作时不需要指定，因为擦除是以块为单位执行，换句话说，擦除时指定的地址从A18开始，这也是为何前面我们称A18~A28为”Block_Address”的原因。

2.2 nandflash操作流程

针对各种操作，nandflash芯片手册上有明确的操作流程说明，以读数据为例，其流程图如下：

对应的时序图有给出，如下，我们可以观察I/Ox对应的那条线，和上图的步骤基本是可以对应起来的。下面分阶段分析一下。

Write 00h

首先，[t1,t2]阶段我们看到有一个00h，对应流程图里的Write 00h。观察对应的CLE和ALE的电平状态，看到CLE为高电平有效，ALE为低电平无效，同时中间WE线还有个从低电平到高电平的跃迁，说明此时是往nandflash芯片写一条命令00h。这个写命令的步骤也是有详细的时序要求的。手册上给出的对应的时序图是命令锁存的时序图(Command Latch Cycle)：

先看CE(Chipset Enable)
该管脚决定是否片选nandflash，低电平有效，只有先选中该nandflash芯片才能操作其他管脚让它工作，操作完成后切换到高电平放弃选中该nandflash芯片，所以确定好选中和去选中之间的时间段，即对相关引脚只需要观察蓝色的t1和t2两条竖线中间部分的时序，两边的其他部分为灰色，无需考虑。

再看CLE(Command Latch Enable)
高电平有效，标识该选中期间的工作模式是向nandflash发送命令，而不是发送地址信息(Address)或者读写数据信息(Data)。这里既然是写命令 那么就一定是 CLE（高电平）ALE（低电平），即针对该时序图，ALE的时序其实我们并不关心。而且我们看到CLE有效和ALE无效的时段和CE有效的时段是基本吻合的。

其中针对CLE时序标识的时间段，tCLS表示CLE建立时间(Command Latch Setup)，对tCLS简单点你可以理解为，我们让CLE引脚变成高电平，总得给人一点时间去变成高电平吧。总不能瞬间就变成高电平了，所以有一个建立(Setup)的过程。tCLH表示CLE信号保持时间(Command Latch Hold)，保持这段时间是确保锁存的数据有效地被nandflash芯片接收。

接下来看WE(Write Enable)
观察图t2和t3两条红线，从t2开始WE变为低电平，使能写操作，虽然 WE是在低电平有效，但并不是说 WE一变成低电平，命令就被锁存了（即nandflash芯片真正获得命令）而是在WE出现一个上升沿时，就像一个触发一样，通知nandflash锁存机制开始工作，经过一段tCLH后命令信息才真正被nandflash芯片锁存并获取到。

最后看I/Ox
真正的命令的内容是通过I/O发送给nandflash芯片的，这里看到命令的数据内容在管脚上也需要保持一段时间，并且也分tDS(Data Setup)阶段和tDH(Data Hold)阶段。

综合下来整个时序图表达的意思就是：当我们希望发送一个命令信息给nandflash芯片时，按时间轴从左往右依次执行如下动作：

  • 从t1开始确保CE使能，选中要操作的nandflash芯片，同时启动CLE，关闭ALE
  • 确保一段时间延时后，即从t1到t2，让上述动作产生的信号稳定后，从t2开始启动写操作，设置WE低电平
  • 然后开始往I/O上发送命令内容，发送的时间点一般要晚于t2，但具体晚多少，似乎芯片不关心。
  • 过一段时间，从t2到t3，几乎和tDS差不多的时间，让I/O管脚电平稳定后，触发WE变为高电平，启动锁存。
  • 给一段时间t3到t4，保持时间(Hold)，确保锁存工作稳定，此时才可以认为nandflash芯片获取了完整的命令内容。

Write Address

然后是[t2,t3]阶段，对应流程图里的Write Address。相应的ALE为高电平有效，CLE为低电平无效，所以操作的是地址，观察WE线，在该时间段内有五次从低到高的跃迁，说明我们向nandlfash依次写入了五个地址序列（对于这五个地址，前面两个是列地址，后面三个是行地址）结合我们前面介绍的该款nandfalsh的对NandFlash的访问机制也是吻合的。
这个写地址的步骤也是有详细的时序要求的。手册上给出的对应的时序图是命令锁存的时序图(Address Latch Cycle)，如下，详细的过程我就不分析了，具体可以参考怎么看时序图–nand flash的读操作详解。

Write 30h

接着[t3,t4]阶段，对应流程图的Write 30h。同第一步，应该是写命令0x30，不赘述了。但是之后数据并未立刻出来，我们看到在Dout N即数据输出之前还有一段时间[t4,t5]，为什么有这段时间？观察R/B这个数据线上说明了原因，这段时间内它是低电平 即指示现在处于忙碌（B）状态，还未准备好数据输出。为什么会这样？答案很简单，因为你写了 一个00h命令，写了要读数据的地址，又写了一个30h命令。 你总要给nandflash芯片一些时间去处理这些命令吧，nandflash芯片为了告诉我们它此刻忙就是通过R/B管脚输出低电平来通知我们这段时间在处理这些命令（实际上是根据命令将你定位的那一页数据读到内部寄存器中），等 R/B变成高电平了，就指示命令处理完毕，这时候我们再去读数据，数据也就可以读出来了。

三、S3C2440的nandflash控制接口

上面讨论的对NandFlash芯片的的访问机制这些分析，都是很底层的操作。其实如果我们使用s3c2440这种高级的处理器的时候，流程还需要我们自己控制，但流程中涉及的时序问题我们根本不需要去自己实现，因为S3C2440这种所谓的SOC芯片上已经集成了一个nandflash控制器单元可以自动完成上面有关时序的所有操作。我们要做的顶多也就往S3C2440定义的和nandflash控制器相关的几个寄存器中设置一下参数配置，然后通过向S3C2440的另外一些和nandflash相关的寄存器中读写一些数据，控制nandflash的流程即可。但对于搞嵌入式开发的工程师来说，还是需要了解一些前面有关时序的问题，原因有二，一是如果你碰到一个没有nandflash控制器的处理器怎么办，那你只能亲自实现这些具体的单元操作细节，同时保证时序正确，其二，在使用象s3c2440这样拥有nandflash控制器的处理器时，考虑到s3c2440需要支持多种nandflash外置芯片，仍然有一些和时序相关的参数需要我们针对特定的nandflash芯片的特性单独配置。

3.1 S3C2440和K9F2G08X0A之间的连线分析

NAND-Flash控制器配置引脚设置

S3C2440手册在NAND Flash Controller章节一开始就指明了外接nandflash存储芯片的管脚配置方法，并指出在芯片复位过程中S3C2440内部的nandflash控制器会检查OM、NCON、GPG13、GPG14和GPG15这五个引脚来确定外接nandflash存储芯片的类型以及各种工艺参数，对应的值如下表所示。其OM引脚用于判断是NORFlash引导还是Nandflash引导，这个在mini2440上是通过开关手动选择的。NCON引脚用于判断外接nandflash的类型，管脚接地(0)表示外接的是每page容量为512Byte的普通类型的nandflash，如果接高电平(1)则是每page容量为2KByte的所谓高级(Advanced)类型的nandflash。GPG13用于区分Page的bit-line的位宽，0表示是16bits，1表示8bits。GPG1用于区分地址周期，即前述根据由于I/O总线宽度一般小于地址总宽度，需要将地址分批发送的批次数。该引脚的设置结果依赖于NCON引脚的设置，当NCON为0时，GPG1设置为0则表示需要3个周期，1则为4个周期；当NCON为1时，GPG1设置为0则表示需要4个周期，1则为5个周期。GPG15表示总线宽度，即nandflash存储芯片的I/O管脚的个数，0为8个，1为16个。

以mini2440搭配的K9F2G08X0A为例，其每page容量为2KByte，即Advance NAND；Page的bit-line的位宽为8，即2Kbytes；地址周期为5；I/O管脚的个数为8。所以相应地，NCON取值应该为1，GPG13取值应该为1、GPG14为1、GPG15为0。

3.2 S3C2440提供的编程接口(寄存器)

S3C2440的nandflash控制器提供了丰富的寄存器接口供我们设置或者读取，封装了对特定的nandflash存储芯片的读写操作具体细节，可以大大减轻我们的编程复杂程度。其提供的寄存器很多，但总的说来我们会主要会操作如下几个寄存器来实现一些特定的操作，譬如

  • 初始化nand flash控制器需要配置寄存器NFCONF
  • 向命令寄存器NFCMD写入操作命令
  • 向地址寄存器NFADDR写入地址
  • 读/写数据前要读取状态寄存器NFSTAT来判断nand flash是否处于忙状态。

下面就针对这些嵌入式开发中常用的主要的寄存器以及其主要的功能总结一下，具体的代码例子主要参考的是mini2440提供的基于tekkaman修改的U-boot(v2009.11)代码中涉及nandflash的代码.

NFCONF寄存器

这个寄存器用于初始化阶段针对外接的特定的nandflash存储芯片对nandflash控制器的工作参数进行配置。其中最重要的有以下三个参数TACL，TWRPH0和TWRPH1。

要想搞清楚这三个参数分别是什么含义，先参考S3C2440手册上有关操作时序的描述图如下：

对比上下两张图：
第一个是t1到t2之间这段时间，对应S3C2440手册的图上就是TACLS，需要保证这段时间足够的长以确保在启动写操作之前使能CE，启动CLE和关闭ALE的动作稳定。
第二个是t2到t3之间这段时间，对应S3C2440手册的图上就是TWRPH0，需要保证这段时间足够的长以确保在触发锁存启动之前I/O管脚上的信号足够稳定。
第三个是t3到t4之间这段时间，对应S3C2440手册的图上就是TWRPH1，需要保证这段时间足够的长以确保锁存工作稳定，从而是nandflash芯片获得正确的命令内容。

总结下来，就是要满足以下的公式：
TACLS = tCLS - tWP
TWRPH0 = tWP
TWRPH1 = tCLH

有关tCLS、tWP以及tCLH的取值自然要参考K9F2G08X0A的数据手册，也就是说我们要根据S3C2440实际连接的nandflash芯片的参数才可以确定TACLS，TWRPH0和TWRPH1的取值。
参考K9F2G08X0A的数据手册，有以下针对发送命令/地址/数据的器件的交流时序特性表(AC(Alternating Current) Timing Characteristics for Command / Address / Data Input)：

根据前面mini2440对K9F2G08X0A的物理连线我们知道输入电压为3.3V，所以
TWRPH0 = tWP，至少要大于12ns
TWRPH1 = tCLH,，至少要大于5ns
TACLS = tCLS - tWP，应该只要确保大于0即可

回来参考S3C2440的用户手册，查看NFCONF寄存器的定义说明如下:

当然这里的时间都是以HCLK为单位的。根据我们在基于mini2440开发板学习S3C2440的系统时钟一文中的描述，HCLK为405MHz。所以，

  • 对TWRPH0，Duration = 1 / (405 10 ^ 6) (TWRPH0 + 1) >= 12 * 10 ^ (-9)。即TWRPH0 >= 3.86，保守地，取TWRPH0至少为4。
  • 对TWRPH1，Duration = 1 / (405 10 ^ 6) (TWRPH1 + 1) >= 5 * 10 ^ (-9)。即TWRPH1 >= 1.025，保守地，取TWRPH1至少为2。
  • 对TACLS，Duration = 1 / (405 10 ^ 6) TACLS > 0。取TACLS至少为1即可，即缺省值。

参考U-boot的源码cpu/arm920t/start.S，在第一阶段将代码从nandflash搬移到SDRAM的过程，就涉及到nandflash的读操作，在具体调用C函数nand_read_ll()操作之前有一段初始化nandflash芯片的汇编，其中一开始就涉及到对NFCONF寄存器的设置：

...
#define NAND_CTL_BASE 0x4E000000

#ifdef CONFIG_S3C2440
/* Offset */
#define oNFCONF 0x00
...

@ reset NAND
mov    r1, #NAND_CTL_BASE
ldr    r2, =( (7<<12)|(7<<8)|(7<<4)|(0<<0) )

ldr    r2, [r1, #oNFCONF]

从上面的代码看出来这里对TACLS、TWRPH0和TWRPH1都取了最大值，有点浪费，但一定管用。另外感觉可以改成如下更好：

ldr    r2, =( (3<<12)|(7<<8)|(7<<4)|(0<<0) )

还有就是为何str后还要ldr一次？

U-boot里面还有一个地方也可以借鉴，就是针对第二阶段驱动nandflash的代码，drivers/mtd/nand/s3c2410_nand.c，里面有个函数board_nand_init()，就是初始化nandflash的地方。针对S3C2440有如下代码：

#if defined(CONFIG_S3C2440)

    twrph1 = 2;

看来这里TWRPH0和TWRPH1的设置和我分析的更接近，但我唯一不解的是TACLS，为啥取0，不过即使这样也一定无问题，因为带入公式后就是-1，换成二进制就是全1，相当于设置了3。

NFCONT寄存器

这个寄存器用于控制nandflash控制器的功能开关。主要功能包括：使能/禁止nandflash控制器(MODE)；使能/禁止片选信号引脚(Reg_nCE)；初始化ECC编解码器(InitECC)。还有其他功能，但在一般应用中用不到，比如锁定NANDflash等。
初始化阶段主要就是设置以上三项，分别使能nandflash控制器，使能片选信号引脚，以及初始化ECC编解码器。

在这点上，cpu/arm920t/start.S和drivers/mtd/nand/s3c2410_nand.c的动作是一致的，以cpu/arm920t/start.S为例：

...
#define NAND_CTL_BASE 0x4E000000

#ifdef CONFIG_S3C2440
...
#define oNFCONT 0x04
...    
    @ reset NAND
    mov    r1, #NAND_CTL_BASE
    ...

    ldr    r2, =( (1<<4)|(0<<1)|(1<<0) )    @ Active low CE Control 
    str    r2, [r1, #oNFCONT]
    ldr    r2, [r1, #oNFCONT]

NFCMMD寄存器

用于发送命令。

依然参考cpu/arm920t/start.S，在初始化阶段，通常我们在第一次操作NAND之前会复位一下NAND flash。复位(Reset)命令的值是ffh。

...
#define NAND_CTL_BASE 0x4E000000

#ifdef CONFIG_S3C2440
/* Offset */
...
#define oNFCMD 0x08
...

    @ reset NAND
    mov    r1, #NAND_CTL_BASE

    ...

    mov    r2, #0xff    @ RESET command
    strb    r2, [r1, #oNFCMD]

NFSTAT寄存器

用于获得nandflash的工作状态，包括片选状态和忙状态。

参考board/tekkamanninja/mini2440/nand_read.c，该文件的操作是针对U-boot的第一阶段初始化阶段，将代码从nandflash搬移到SDRAM的过程，就涉及到nandflash的读操作。有一个nand_wait()函数，在读/写数据前需要调用该函数读取状态寄存器NFSTAT的第0位来判断nand flash当前是否处于忙状态。

...
#define NF_BASE        0x4e000000
#if defined(CONFIG_S3C2410)
...
#elif defined(CONFIG_S3C2440) || defined(CONFIG_S3C2442)
...
#define NFSTAT        __REGb(NF_BASE + 0x20)
#define NFSTAT_BUSY    1
...

static inline void nand_wait(void)
{
    int i;

    while (!(NFSTAT & NFSTAT_BUSY))
        for (i=0; i<10; i++);
}

NFADDR寄存器

用于发送地址。

参考board/tekkamanninja/mini2440/nand_read.c，该文件的操作是针对U-boot的第一阶段初始化阶段，将代码从nandflash搬移到SDRAM的过程，就涉及到nandflash的读操作。其中函数nand_read_page_ll()就是读一个page，我们知道对于NANDflash读写都是以page为单位操作。

注：我对该段代码针对S3C2440按照NFDATA16处理有点疑问，我的mini2440配的nandflash不是8位输出的吗，怎么可以按照16位NFDATA16处理呢。
针对这个问题，我后来实际测试了一下，发现对于我的mini2440所搭配的K9F2G08U0B，采用NFDATA的方式，即8位的方式读也是OK的，总之不是很理解为何tekkaman的代码里对S3C2440的芯片采用一次读16位的方式，感觉这个应该要和具体nandflash芯片的bit-lines的宽度匹配才好。

...
#define NF_BASE        0x4e000000
#if defined(CONFIG_S3C2410)
...
#elif defined(CONFIG_S3C2440) || defined(CONFIG_S3C2442)
...
#define NFADDR        __REGb(NF_BASE + 0xc)
#define NFDATA        __REGb(NF_BASE + 0x10)
#define NFDATA16    __REGw(NF_BASE + 0x10)
...
#endif

static int nand_read_page_ll(struct boot_nand_t * nand, unsigned char *buf, unsigned long addr)
{
    unsigned short *ptr16 = (unsigned short *)buf;
    unsigned int i, page_num;

    nand_clear_RnB();

    NFCMD = NAND_CMD_READ0;

    if (nand->page_size == 512) {
        /* Write Address */
        NFADDR = addr & 0xff;
        NFADDR = (addr >> 9) & 0xff;
        NFADDR = (addr >> 17) & 0xff;
        NFADDR = (addr >> 25) & 0xff;
    } else if (nand->page_size == 2048) {
        page_num = addr >> 11; /* addr / 2048 */
        /* Write Address */
        NFADDR = 0;
        NFADDR = 0;
        NFADDR = page_num & 0xff;
        NFADDR = (page_num >> 8) & 0xff;
        NFADDR = (page_num >> 16) & 0xff;
        NFCMD = NAND_CMD_READSTART;
    } else {
        return -1;
    }
    nand_wait();

#if defined(CONFIG_S3C2410)
    for (i = 0; i < nand->page_size; i++) {
        *buf = (NFDATA & 0xff);
        buf++;
    }
#elif defined(CONFIG_S3C2440) || defined(CONFIG_S3C2442)
    for (i = 0; i < (nand->page_size>>1); i++) {
        *ptr16 = NFDATA16;
        ptr16++;
    }
#endif

    return nand->page_size;
}
