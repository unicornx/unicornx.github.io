针对mini2440开发板的U-boot移植笔记(v2009.11)

发表于 2015-08-19   |   分类于 Bootloader   |  

文章系原创，如需转载请注明：转载自”Blog of UnicornX” (http://unicornx.github.io/)

最新更新于：2015-09-24

总体目标

网上看了很多高手移植U-boot到mini2440的帖子，但发现大部分就是一个过程的流水账，跟着做一趟下来，思路不是很明确。所以最后还是痛下决心，按照自己的理解，从需求出发，以更贴近实战的方式，分步骤，一步一步实现一次比较完整的移植。

首先确定一下我的目标，这里的移植总的参考蓝本还是以友善之臂提供的Mini2440之U-boot使用及移植详细手册2010-4-16为主要蓝本，以tekkaman的2009.11的U-boot移植代码为主要代码参考（注：本文的代码修改基本上就是参考tekkaman的2009.11版本，其中除了明显觉得是错误的地方做了修改，其他地方，比如可以那么改，不是这么改的，计划等这次熟悉了以后再按自己的思路做一遍）。功能上做了部分裁剪，没有实现那么多。主要是期望实现一个具备以下功能点的u-boot loader:

  • 支持串口显示
  • 支持nandflash启动，支持nandflash的一些基本操作维护命令
  • 支持usb传输并在nandflash上烧写u-boot/kernel/rootfs,（注：对rootfs的大小目前有限制，不可以超过STACK_BASE/0x33F00000-PHYS_SDRAM_1/0x30000000 = 0x3F00000 = 63MB
  • 支持通过U-boot引导nandflash上的内核
  • (可选)支持norflash启动，支持norflash的一些基本操作维护命令，目前暂未实现。

涉及的知识点：

  • execption/interrupt
  • clock
  • gpio/led
  • sdram
  • uart
  • nand-flash
  • usb（涉及DMA）
  • nor-flash （可选）

大致的计划步骤如下：

步骤 目标
Step1 建立基本框架，为后继工作打下基础，仅保证编译通过
Step2 确保第一阶段运行正常。通过Norflash启动后利用supervivi对SDRAM初始化，然后通过supervivi将程序下载到sdram上运行，通过点灯确认第一阶段步骤正常
Step3 仍然针对第一阶段，但在这一步使能代码重定向，利用重定向将代码从nandflash搬移到SDRAM自动运行，通过点灯确认第一阶段步骤正常，并可以跳转到第二阶段的入口处。
Step4 确保代码重定向后第二阶段的基本流程正确，通过点灯验证start_armboot基本流程正确，添加UART打印功能，为后面更复杂的功能调试做准备
Step5 为第二阶段添加nandflash功能
Step6 添加usb传输功能
Step7 在step5和step6的基础上从nandflash引导内核

对应的代码我已经上传到https://git.oschina.net，可以使用如下命令获得

$ git clone https://git.oschina.net/unicornx/mini2440.git
$ git checkout uboot-200911-1st-try

Step1.2 建立开发板mini2440的目录

根据对U-boot的平台定制机制的理解(基于v2009.11)的总结，相应的，对mini2440的平台定制策略大致如下：

定制层次 VAR value 对应的目录
架构 ARCH arm lib_arm，已存在，有待修改
处理器核 CPU arm920t cpu/arm920t，已存在，有待修改
集成芯片 SOC s3c24x0 cpu/arm920t/s3c24x0，已存在，有待修改
目标板 VENDOR + BOARD tekkamanninja + mini2440 board/tekkamanninja/mini2440，需要新增，注：如果VENDOR设置为NULL，则目录结构为board/mini2440

所以建立开发板mini2440的目录board/tekkamanninja/mini2440，并复制sbc2410x的文件到此，做适当修改。目的：以sbc2410x 为蓝本，加快移植进度。
顺便说一下，有关board/sbc2410x和board/samsung/smdk2410以前也没太去区分，这次上网查了一下，原来sbc2410x是广州友善之臂设计生产的一款基于三星ARM处理器S3C2410X的开发板，而smdk2410则是三星自己设计的一款基于S3C2410的开发板，SMDK就是Samsung MCU Development Kit的意思。mini2440也是广州友善之臂设计生产的一款基于三星ARM处理器S3C2440的开发板，难怪mini2440的U-boot手册首先选择基于sbc2410x为蓝本了。不过我略微多心扫了一下U-boot-2013.01版本，似乎sbc2410x被移除了。是不是以后就拷贝board/samsung/smdk2410做蓝本了呢

$ cd tekkamanninja/mini2440
$ mv sbc2410x.c mini2440.c

Step2 确保第一阶段运行正常

先总结一下mini2440的启动部分的代码大致逻辑如下，总体上分为两个阶段：

@cpu/arm920t/start.S
_start:    b start_code
... //异常向量表
//第一阶段
start_code: 
{
    1.1> 让系统进入SVC（管理员模式）
    1.2> 关闭看门狗
    1.3> 关闭所有中断
    1.4> 设置时钟的分频比
    1.5> cpu_init_crit
    {
        1.5.1> 关闭MMU和cache    
        1.5.2> lowlevel_init@board/tekkamanninja/mini2440
        {
            1.5.2.1> 初始化SDRAM
        }
    }
    1.6> 代码重定向，如果不是在内存中(一般是在NOR-Flash里，就将自己从NOR-Flash拷贝到SDRAM中
    1.7> 堆栈初始化，为第二阶段的C语言做准备
    1.8> 对BSS段清零，为第二阶段的C语言做准备
    1.9> 跳向第二阶段的C语言代码入口_start_armboot，该部分代码已经被重定向到内存
}

@lib_arm/board.c
//第二阶段
void start_armboot(void)
{
    ...
}

由于该Step的目标主要是要确保第一阶段的主体逻辑运行正常，为简化该阶段的工作，尽量不涉及外设的操作：

  • 可以屏蔽第一阶段中的代码重定向，所以不需要使能nandflash和SDRAM
  • 屏蔽第二阶段
  • 仅使用点灯的方式确认程序运行状态，所以不涉及串口终端输出，即不涉及UART

该过程中应该会涉及

  • 中断处理，仅仅是初始化为关闭
  • 时钟的初始化设置，必须的。
  • GPIO/LED点灯，用于调试

综上所述，由于该阶段不使能SDRAM，所以可以采用如下手段之一进行测试ﺿ
手段一：先选择Norflash启动后，使用supervivi将编译后的U-boot程序通过DWN烧写到nandflash中，切换为nandflash启动，上电后S3C2440内部机制会将nandflash的前4KB自动拷贝到内部的SRAM中执行，利用点灯查看第一阶段主体逻辑运行状况
手段二：Norflash启动后，使用supervivi对SDRAM初始化，然后通过supervivi和DWN将我的处于开发阶段的uboot下载到sdram上运行，同样利用点灯查看第一阶段主体逻辑运行状况

Step2.4 测试

先用手段一：Norflash启动后，使用supervivi将编译后的U-boot程序通过DWN烧写到nandflash中。然后切换为nandflash启动，上电后S3C2440内部机制会将nandflash的前4KB自动拷贝到内部的SRAM中执行，该种测试条件下只会看到LED1亮，LED2不亮，原因很简单，由于此时SDRAM还未初始化也没有执行代码重定位动作，所以在start.S中执行到ldr pc, _start_armboot语句后跳转到SDRAM的TEXT_BASE处无法获得代码，所以不会点亮LED2

再用手段二：Norflash启动后，使用supervivi对SDRAM初始化，然后通过supervivi和DWN将我的处于开发阶段的uboot下载到sdram上运行，supervivi菜单选d，下载地址选择TEXT_BASE，即0x33f80000。这时应该可以看到LED1和LED2都亮了。其实可以基于该手段的思想，直接在SDRAM上调试U-boot的第二阶段，这样就用不着每次都烧写nandflash了。

Step3 使能代码重定向

在该步骤，摆脱对S3C2440的stepstone以及supervivi初始化内存操作的依赖，自己初始化SDRAM，使能代码重定向，利用重定向自己将代码从nandflash搬移到SDRAM运行。当然这里仍然要依赖supervivi将U-boot烧写到nandflash的指定区域。

测试方法：在start_armboot函数里打开点灯，从s3c2440的内部SRAM里跳转到SDRAM里运行start_armboot，看到灯被点亮即可

涉及(在上一步的基础上)

  • SDRAM初始化
  • nandflash基本初始化nand_read
  • 代码重定向

再出发

做完以上步骤后，我考虑了一下感觉结合mini2440提供的supervivi调试移植U-boot有一个更好的流程。与前面第一种方案的主要区别在于将第一阶段的实现放到最后来做，先移植调试第二阶段。即采用norflash+supervivi启动，然后将调试的U-boot下载到SDRAM中运行，调试直到可以引导内核。最后再调试搬运。

调整后移植的步骤如下：

步骤 目标
Step1 建立基本框架，为后继工作打下基础，仅保证编译通过
Step2 通过Norflash启动后利用supervivi对SDRAM初始化，然后通过supervivi将程序下载到sdram上运行，屏蔽第一阶段，通过点灯确认可以跳转到第二阶段的入口处。
Step3 通过点灯验证start_armboot基本流程正确，添加UART打印功能，为后面更复杂的功能调试做准备
Step4 为第二阶段添加nandflash功能
Step5 为第二阶段添加usb传输功能
Step6 在step5和step6的基础上从nandflash引导内核
Step7 使能代码重定向，利用重定向将代码从nandflash搬移到SDRAM自动运行第二阶段，至此完成。
