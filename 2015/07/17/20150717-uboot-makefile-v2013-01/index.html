理解u-boot的Makefile(基于v2013.01)

发表于 2015-07-17   |   分类于 Bootloader   |  

文章系原创，如需转载请注明：转载自”Blog of UnicornX” (http://unicornx.github.io/)

最新更新于：2015-7-20

参考：
u-boot2013.01.01 for s5pv210：《Makefile分析》

总体上看2013.01版本的主Makefile文件相较于2009.11版本并没有本质上的改变。

规则描述前的准备工作

设置版本

VERSION = 2013
PATCHLEVEL = 01
SUBLEVEL =
EXTRAVERSION =
ifneq "$(SUBLEVEL)" ""
U_BOOT_VERSION = $(VERSION).$(PATCHLEVEL).$(SUBLEVEL)$(EXTRAVERSION)
else
U_BOOT_VERSION = $(VERSION).$(PATCHLEVEL)$(EXTRAVERSION)
endif

这里都是些有关版本号的配置，参考README的“Versioning:”章节，u-boot对其版本的定义方式如下：

Versioning:
===========
Starting with the release in October 2008, the names of the releases
were changed from numerical release numbers without deeper meaning
into a time stamp based numbering. Regular releases are identified by
names consisting of the calendar year and month of the release date.
Additional fields (if present) indicate release candidates or bug fix
releases in "stable" maintenance trees.

Examples:
    U-Boot v2009.11        - Release November 2009
    U-Boot v2009.11.1   - Release 1 in version November 2009 stable tree
    U-Boot v2010.09-rc1 - Release candiate 1 for September 2010 release

相比较于2009.11版本的README，这里的描述清晰了很多。大致的意思是说从2008年10月起，发行版本的名称格式及其含义发生了变化,从没有什么具体含义的数字版本号变为基于时间戳的数字作为版本号。具体的格式为：

$(VERSION).$(PATCHLEVEL).$(SUBLEVEL)$(EXTRAVERSION

VERSION对应发布的年，PATCHLEVEL对应发布的月，该两项不可以为空。
SUBLEVEL对应子版本号，即某固定年月的第几次小版本发布，EXTRAVERSION对应额外的信息，譬如指明为候选发布版本号等，该两项可能为空。
具体例子见README。

TIMESTAMP_FILE = $(obj)include/generated/timestamp_autogenerated.h
VERSION_FILE = $(obj)include/generated/version_autogenerated.h

设置主机host的编译环境

因为是嵌入式开发，采用的是交叉编译，所以需要设置主机的编译环境，包括：获取主机的arch和os，以及设置shell。
sed -e的意思，选项允许在同一行里执行多条命令，就是表示后面跟的是一串命令脚本，s/abc/def/的命令表达式，就是表示要从uname -m的输出中，查找到内容为abc的，然后替换成def

HOSTARCH := $(shell uname -m | \
    sed -e s/i.86/x86/ \
        -e s/sun4u/sparc64/ \
        -e s/arm.*/arm/ \
        -e s/sa110/arm/ \
        -e s/ppc64/powerpc/ \
        -e s/ppc/powerpc/ \
        -e s/macppc/powerpc/\
        -e s/sh.*/sh/)

HOSTOS := $(shell uname -s | tr '[:upper:]' '[:lower:]' | \
        sed -e 's/\(cygwin\).*/cygwin/')

# Set shell to bash if possible, otherwise fall back to sh
SHELL := $(shell if [ -x "$$BASH" ]; then echo $$BASH; \
    else if [ -x /bin/bash ]; then echo /bin/bash; \
    else echo sh; fi; fi)

export相当于C语言中的extern，把变量输出到全局，这样这些变量将设置为环境变量
注意:export的语意就是将自定义变量转成环境变量,在系统的任何地方你都可以使用这个变量!

export    HOSTARCH HOSTOS SHELL

# Deal with colliding definitions from tcsh etc.
VENDOR=

处理make命令行的选项参数

s选项

检查是否禁止在编译过程中向终端输出信息，利用findstring函数检查变量 MAKEFLAGS 是否为参数’s’
当编译U-boot的时候，我们会输入make [选项] [目标] …如果输入:make -s，则MAKEFLAGS = s
XECHO = echo 编译时将输出编译过程的信息，否则安静编译,不输出信息

#########################################################################
# Allow for silent builds
ifeq (,$(findstring s,$(MAKEFLAGS)))
XECHO = echo
else
XECHO = :
endif

O选项

uboot允许将生成的所有编译结果都存放在源代码主目录以外的一个单独的目录下，这样就可以保持源代码目录比较干净，这个目录用变量BUILD_DIR来表示，缺省为空，即为uboot源代码目录的主目录。这个变量可以修改，具体的设置方法，可以参考Makefile中的注释，大致有两种方法可以实现：
1) 指定编译参数 O=目录，该选项会覆盖BUILD_DIR的值，例如: 执行make O= /tmp/build all后，BUILD_DIR的值就是/tmp/build all。
2) 设置环境变量BUILD_DIR，该变量指定目录并设置为设置为环境变量，例如:export BUILD_DIR=/tmp/build，然后再make。该方法也适用于执行MAKEALL，MAKEALL主要用于u-boot自身的测试发布工作。
我们自己编译时一般不需要指定，就在当前源代码顶层目录下生成，所以直接make就好。 

#########################################################################
#
# U-boot build supports producing a object files to the separate external
# directory. Two use cases are supported:
#
# 1) Add O= to the make command line
# 'make O=/tmp/build all'
#
# 2) Set environement variable BUILD_DIR to point to the desired location
# 'export BUILD_DIR=/tmp/build'
# 'make'
#
# The second approach can also be used with a MAKEALL script
# 'export BUILD_DIR=/tmp/build'
# './MAKEALL'
#
# Command line 'O=' setting overrides BUILD_DIR environent variable.
#
# When none of the above methods is used the local build is performed and
# the object files are placed in the source directory.
#

ifdef O
ifeq ("$(origin O)", "command line")
BUILD_DIR := $(O)
endif
endif

下面是一系列和创建build目录相关的动作，如果创建失败，就输出错误并退出。注：这里if用了一个三段式的形式，#if(a,b,c)这样的形式，执行步骤为，先判断a的真假，如果为真，则执行b，如果为假，则执行c。

ifneq ($(BUILD_DIR),)
saved-output := $(BUILD_DIR)

# Attempt to create a output directory.
$(shell [ -d ${BUILD_DIR} ] || mkdir -p ${BUILD_DIR})

# Verify if it was successful.
BUILD_DIR := $(shell cd $(BUILD_DIR) && /bin/pwd)
$(if $(BUILD_DIR),,$(error output directory "$(saved-output)" does not exist))
endif # ifneq ($(BUILD_DIR),)

定义几个有用的环境变量：

其中：

  • OBJTREE： 如果指定了编译目录BUILD_DIR的值，就采用$(BUILD_DIR)替换它，否则就等于$(CURDIR)，CURDUR是Make自定义的变量，指当前工作路径，因为我们在编译make之前都要cd到uboot源代码的主目录下，所以其值就等于uboot源代码的主目录的文件系统路径。
  • SRCTREE：设置为$(CURDIR)

其他的就不多啰嗦了。

OBJTREE        := $(if $(BUILD_DIR),$(BUILD_DIR),$(CURDIR))
SPLTREE        := $(OBJTREE)/spl
SRCTREE        := $(CURDIR)
TOPDIR        := $(SRCTREE)
LNDIR        := $(OBJTREE)
export    TOPDIR SRCTREE OBJTREE SPLTREE

MKCONFIG    := $(SRCTREE)/mkconfig
export MKCONFIG

ifneq ($(OBJTREE),$(SRCTREE))
REMOTE_BUILD    := 1
export REMOTE_BUILD
endif

仔细看看下面这段注释，obj和src也是两个编译中重要的变量，虽然这两个变量主要是定义在顶层目录的config.mk文件里(config.mk也是一个相当重要的文件，具体我们后面会专门讨论)，但由于主Makefile中有些伪规则，譬如unconfig, clean, clobber, distclean等会用到它们，所以预先定义在这里。如果我们指定了BUILD_DIR，则$(OBJTREE)和$(SRCTREE)的值会不同，那么obj和src会相应地被赋予$(OBJTREE)和$(SRCTREE)的值，否则就都是为空，即缺省为当前主目录。

# $(obj) and (src) are defined in config.mk but here in main Makefile
# we also need them before config.mk is included which is the case for
# some targets like unconfig, clean, clobber, distclean, etc.
ifneq ($(OBJTREE),$(SRCTREE))
obj := $(OBJTREE)/
src := $(SRCTREE)/
else
obj :=
src :=
endif
export obj src

# Make sure CDPATH settings don't interfere
unexport CDPATH

至此，前期准备工作就绪，Makefile文件从这里开始往下就是make的规则描述了，一个Makefile一般会包括很多规则，U-boot的规则整体上由三部分组成：

  • 生成u-boot镜像的终极规则及其所依赖的规则

  • 生成针对各种开发板的配置开关项列表的规则

  • 其他辅助规则

下面我们依次分别来看一下。

生成u-boot最终img的主要规则

先还是一些准备工作，SUBDIRS这个变量会保存参与编译的所有子目录，即参与编译的模块的名称，用空格分隔，make时会按照SUBDIRS中子目录出现的顺序，依次进入并制作。根据注释，因为make时会需要用到一些uboot自制的tools，所以将tools目录列为SUBDIRS的第一个成员，然后会根据一些编译的配置选项对这个SUBDIRS变量追加别的子目录。

#########################################################################

# The "tools" are needed early, so put this first
# Don't include stuff already done in $(LIBS)
# The "examples" conditionally depend on U-Boot (say, when USE_PRIVATE_LIBGCC
# is "yes"), so compile examples after U-Boot is compiled.
SUBDIR_TOOLS = tools
SUBDIR_EXAMPLES = examples/standalone examples/api
SUBDIRS = $(SUBDIR_TOOLS)

.PHONY : $(SUBDIRS) $(VERSION_FILE) $(TIMESTAMP_FILE)

一个重要的判断

ifeq ($(obj)include/config.mk,$(wildcard $(obj)include/config.mk))

上面这句ifeq判断的话很重要，首先这句话是一个if判断，在判断什么呢？wildcard的格式是wildcard PATTERN，PATTERN表达式一般是个文件路径，wildcard会根据PATTERN表达式匹配并查找该文件是否存在，如果存在就将其路径字符串返回，否则就返回空，所以可以用wildcard来查找一个文件是否存在。这里就是在判断“$(obj)include/config.mk”这个文件是否存在。假设我们没有指定编译输出目录，则$(obj为空，所以就是在检查“include/config.mk”这个文件是否存在，这个文件是我们在编译之前执行形如make xxx_config的命令创建出来的，所以通过判断该文件是否存在可以知道我们在make之前是否执行了make xxx_config，如果还没有执行make xxx_config进行相应配置就执行make这是不允许的。有关make xxx_config干了些什么，参考理解u-boot的make xxx_config机制(基于v2009.11)

注意这个ifeq其对应的endif在下面很远的地方，具体是在主Makefile文件的第719行一个重要判断的else分支，而且还是个二路分支判断语句，去掉内部的大段逻辑，总框架如下：

ifeq ($(obj)include/config.mk,$(wildcard $(obj)include/config.mk)) 
# 分支一，可编译最终结果相关的一系列规则描述，此处跳过300+行
... 
else    # !config.mk 第719行
# 分支二： 
all $(obj)u-boot.hex $(obj)u-boot.srec $(obj)u-boot.bin \
$(obj)u-boot.img $(obj)u-boot.dis $(obj)u-boot \
$(filter-out tools,$(SUBDIRS)) \
updater depend dep tags ctags etags cscope $(obj)System.map:
    @echo "System not configured - see README" >&2
    @ exit 1

tools: $(VERSION_FILE) $(TIMESTAMP_FILE)
    $(MAKE) -C $@ all
endif    # config.mk

所以这个判断的作用就是确保在make或者make all之前我们必须针对你需要移植的开发板运行过make xxx_config。判断的原则就是看$(obj)include/config.mk这个文件是否存在，如果存在说明运行过配置，就允许继续make，否则会打印错误告警如下：

fs@ubuntu$ make
System not configured - see README
make: *** [all] Error 1

所以我们重点看分支一，走进分支一就是在具体执行制作u-boot二进制镜像的编译工作了，主要规则都在这里了。

导入(include)重要的配置文件(.mk)

导入$(obj)include/autoconf.mk

这里出现的all是个假的终极目标，我们输入make all时并不是运行它，因为这个all既没有依赖，也没有规则，它下面的sinclude语句不是它的动作，原因很简单，因为前面没有TAB。这个all的用途在这里的注释里说得很清楚了，定义这个伪目标all的目的是防止autoconf.mk.dep中的依赖目标成为默认终极目标，因为默认make会把主Makefile里碰到的第一个目标当作终极目标。

# Include autoconf.mk before config.mk so that the config options are available
# to all top level build files.  We need the dummy all: target to prevent the
# dependency target in autoconf.mk.dep from being the default.
all:

所以make接着开始执行sinclude $(obj)include/autoconf.mk.dep，参考Makefile中include、-include、sinclude的区别,总结如下：

  • 使用“include FILENAMES…”，make程序处理时，如果“FILENAMES”列表中的任何一个文件不能正常读取而且不存在一个创建此文件的规则时make程序将会提示错误并退出。
  • 使用“-include FILENAMES…”的情况是，当所包含的文件不存在或者不存在一个规则去创建它，make程序会继续执行，只有真正由于不能正确完成终极目标的重建时（某些必需的目标无法在当前已读取的makefile文件内容中找到正确的重建规则），才会提示致命错误并退出。
  • 为了和其它的make程序进行兼容。也可以使用“sinclude”来代替“-include”（GNU所支持的方式）。

autoconf.mk和autoconf.mk.dep这些文件是在后继规则中被创建出来的，创建autoconf.mk的具体的位置。如果是第一次干净的build，还没有被创建，所以”sinclude”的意思就是如果这些文件还不存在就忽略这两条命令，继续。

sinclude $(obj)include/autoconf.mk.dep
sinclude $(obj)include/autoconf.mk

ifndef CONFIG_SANDBOX
SUBDIRS += $(SUBDIR_EXAMPLES)
endif

导入$(obj)include/config.mk

这时候$(obj)include/config.mk被导入，这个文件是通过运行make xxx_config产生的。这一步必须成功也一定会成功，因为根据前述ifeq的逻辑，逻辑走到这里说明已经做过make xxx_config。

注意这里执行的导出ARCH，CPU，BOARD，VENDOR，SOC的行为。这些记录在$(obj)include/config.mk中的变量的值会直接导致后继导入的$(TOPDIR)/config.mk和生成autoconf.mk文件时代码中所有引用这些变量的地方执行字符串替换，从而使相关的配置只针对make第一阶段，即make xxx_config针对的特定开发板。详细的操作参考理解u-boot的顶层config.mk(基于v2013.01)的”导入各级配置文件(.mk)，实现平台的定制化”章节。

# load ARCH, BOARD, and CPU configuration
include $(obj)include/config.mk
export    ARCH CPU BOARD VENDOR SOC

判断如果宿主机的架构和目标板的架构是同一种架构，则CROSS_COMPILE为空，也即采用宿主机默认的交叉编译链。通常我们将目标板使用的交叉工具链安装好以后，在宿主机中设置环境变量，指定开发板使用的交叉编译链。

# set default to nothing for native builds
ifeq ($(HOSTARCH),$(ARCH))
CROSS_COMPILE ?=
endif

在执行make命令时指定CROSS_COMPILE，有如下两种方法：

  • 方法一，在调用make命令时直接指定，譬如：

    $ make all CROSS_COMPILE=arm-linux-gnueabi-

  • 方法二：调用make命令前先定义该变量，再make，譬如：

    $ export CROSS_COMPILE=arm-linux-gnueabi-
$ make all

如果不指定就认为是本地编译。

导入$(TOPDIR)/config.mk

注：这里include是uboot顶层目录下的config.mk文件。这也是一个重要的make配置文件，参考理解u-boot的顶层config.mk(基于v2013.01)。

# load other configuration
include $(TOPDIR)/config.mk

搜索链接脚本文件

仔细读一下下面的这段注释，意思大致是说用户自己可以通过定义LDSCRIPT或者CONFIG_SYS_LDSCRIPT来指定lds脚本文件。如果指定了则makefile就会用，如果这两个变量都为空，则makefile会根据特定的规则在一些标准位置上搜索lds文件。

# If board code explicitly specified LDSCRIPT or CONFIG_SYS_LDSCRIPT, use
# that (or fail if absent).  Otherwise, search for a linker script in a
# standard location.

LDSCRIPT_MAKEFILE_DIR = $(dir $(LDSCRIPT))

先使用dir函数尝试从LDSCRIPT中取出目录部分。

如果LDSCRIPT没定义再看看用户有没有自己通过CONFIG_SYS_LDSCRIPT指定。注意：如果CONFIG_SYS_LDSCRIPT有定义也是在include/autoconf.mk中定义。

ifndef LDSCRIPT
    #LDSCRIPT := $(TOPDIR)/board/$(BOARDDIR)/u-boot.lds.debug
    ifdef CONFIG_SYS_LDSCRIPT
        # need to strip off double quotes
        LDSCRIPT := $(subst ",,$(CONFIG_SYS_LDSCRIPT))
    endif
endif

如果既没有定义LDSCRIPT，也没有配置CONFIG_SYS_LDSCRIPT，接下来会按照如下顺序进行搜索:
如果支持uboot从nandflash启动(CONFIG_NAND_U_BOOT的值为’y’)则先尝试指定lds脚本为$(TOPDIR)/board/$(BOARDDIR)/u-boot-nand.lds，如果该文件不存在，则尝试查找$(TOPDIR)/$(CPUDIR)/u-boot-nand.lds。
如果还是没找到，则尝试$(TOPDIR)/board/$(BOARDDIR)/u-boot.lds。
如果还是没找到，则尝试$(TOPDIR)/$(CPUDIR)/u-boot.lds。
如果还是没找到，则尝试$(TOPDIR)/arch/$(ARCH)/cpu/u-boot.lds。
如果还是没找到，则认为失败，输出could not find linker script。

# If there is no specified link script, we look in a number of places for it
ifndef LDSCRIPT
    ifeq ($(CONFIG_NAND_U_BOOT),y)
        LDSCRIPT := $(TOPDIR)/board/$(BOARDDIR)/u-boot-nand.lds
        ifeq ($(wildcard $(LDSCRIPT)),)
            LDSCRIPT := $(TOPDIR)/$(CPUDIR)/u-boot-nand.lds
        endif
    endif
    ifeq ($(wildcard $(LDSCRIPT)),)
        LDSCRIPT := $(TOPDIR)/board/$(BOARDDIR)/u-boot.lds
    endif
    ifeq ($(wildcard $(LDSCRIPT)),)
        LDSCRIPT := $(TOPDIR)/$(CPUDIR)/u-boot.lds
    endif
    ifeq ($(wildcard $(LDSCRIPT)),)
        LDSCRIPT := $(TOPDIR)/arch/$(ARCH)/cpu/u-boot.lds
        # We don't expect a Makefile here
        LDSCRIPT_MAKEFILE_DIR =
    endif
    ifeq ($(wildcard $(LDSCRIPT)),)
$(error could not find linker script)
    endif
endif

定义制作过程中需要的对象

定义目标变量OBJS

start.o是必须放在第一位的，顺序很重要,没事别乱改这些目标对象和库的顺序!
其中CPUDIR=arch/$(ARCH)/cpu/$(CPU)，定义在源码顶层目录的config.mk中

#########################################################################
# U-Boot objects....order is important (i.e. start must be first)

OBJS  = $(CPUDIR)/start.o
ifeq ($(CPU),x86)
RESET_OBJS-$(CONFIG_X86_NO_RESET_VECTOR) += $(CPUDIR)/start16.o
RESET_OBJS-$(CONFIG_X86_NO_RESET_VECTOR) += $(CPUDIR)/resetvec.o
endif
ifeq ($(CPU),ppc4xx)
OBJS += $(CPUDIR)/resetvec.o
endif
ifeq ($(CPU),mpc85xx)
OBJS += $(CPUDIR)/resetvec.o
endif

OBJS := $(addprefix $(obj),$(OBJS) $(RESET_OBJS-))

定义库变量LIBS

LIBS变量指明了U-boot需要的库文件，包括平台/开发板相关的目录、通用目录下相应的库，都通过相应的子目录编译得到。
但新版本的Makefile里并不是直接对LIBS变量赋值，而是先对中间变量LIBS-y和LIBS-n进行赋值。其中使用了如下技巧：
假设一个变量FOO(大部分形如CONFIG_xxx)，其取值无非是y或者n
则语句：LIBS-$(FOO) += libxxxx.o的效果会根据$(FOO)的取值是y还是n，将libxxxx.o分别添加到LIBS-y或者LIBS-n中。而最后LIBS变量的值是从LIBS-y里取，达到的效果就是可以根据FOO的取值决定某个lib库文件是否参与编译。

HAVE_VENDOR_COMMON_LIB = $(if $(wildcard board/$(VENDOR)/common/Makefile),y,n)

LIBS-y += lib/libgeneric.o
LIBS-y += lib/lzma/liblzma.o
LIBS-y += lib/lzo/liblzo.o
LIBS-y += lib/zlib/libz.o
LIBS-$(CONFIG_TIZEN) += lib/tizen/libtizen.o
LIBS-$(HAVE_VENDOR_COMMON_LIB) += board/$(VENDOR)/common/lib$(VENDOR).o
LIBS-y += $(CPUDIR)/lib$(CPU).o
ifdef SOC
LIBS-y += $(CPUDIR)/$(SOC)/lib$(SOC).o
endif
ifeq ($(CPU),ixp)
LIBS-y += drivers/net/npe/libnpe.o
endif
LIBS-$(CONFIG_OF_EMBED) += dts/libdts.o
LIBS-y += arch/$(ARCH)/lib/lib$(ARCH).o
LIBS-y += fs/libfs.o \
    fs/cbfs/libcbfs.o \
    fs/cramfs/libcramfs.o \
    fs/ext4/libext4fs.o \
    fs/fat/libfat.o \
    fs/fdos/libfdos.o \

u-boot ELF格式的U-boot文件

System.map U-Boot文件的符号表，System.map是U-Boot的符号表，它包含了U-Boot的全局变量和函数的地址信息，也就是将nm命令查看u-boot的输出信息经过过滤和排序后输出到System.map。System.map表示的是地址标号到该标号表示的地址的一个映射关系。System.map每一行的格式都是“addr type name”，addr是标号对应的地址值，name是标号名，type表示标号的类型。U-Boot的编译和运行并不一定要生成System.map，这个文件主要是提供给用户或外部程序调试时使用的。

u-boot依赖于以下目标

依赖的目标 制作该目标的动作
depend 对于$(SUBDIRS)，cpu/$(CPU)，$(dir $(LDSCRIPT))中的每个元素都进入该目录执行“make _depend”，生成各个子目录的.depend文件，.depend列出每个目标文件的依赖文件。
仅仅有目标和依赖，没有命令的，所以调用该目标时是通过隐含规则执行隐含命令的。
注意depend目标仅仅是生成了文件.depend，但是并没有执行生成任何的目标文件（比如start.o）二进制文本的任何命令，也就是说在顶层目录make depend后，在cpu/$(CPU)目录下并没有start.o文件
$(SUBDIRS) 执行tools ，examples/standalone ，examples/api目录下的Makefile
$(OBJS) 对于OBJS包含的每个成员，都进入cpu/$(CPU)目录（即cpu/arm920t）编译它们。
$(LIBBOARD) 查看对应规则
$(LIBS) 对于LIBS中的每个成员，都进入相应的子目录执行“make”命令编译它们。例如对于LIBS中的“common/libcommon.a”成员，程序将进入common目录执行Makefile，生成libcommon.a
$(LDSCRIPT) 注意对于versatile开发板，$(LDSCRIPT)是定于在lib_arm/config.mk里，LDSCRIPT := $(SRCTREE)/cpu/$(CPU)/u-boot.lds所以是cd到cpu/arm926ejs/目录下，执行make u-boot.lds命令
$(obj)u-boot.lds 将cpu/arm926ejs/u-boot.lds经cpp预处理后输出到U-Boot顶层目录下的u-boot.lds文件。

生成autoconf.mk的规则

autoconf.mk文件是一个很重要的有关配置的文件。这个文件是在make过程中自动生成的。打开autoconf.mk文件看看，就会看到形如下的内容：

CONFIG_SYS_GBL_DATA_SIZE=128
CONFIG_SYS_LONGHELP=y
CONFIG_SYS_LOAD_ADDR="0x7fc0"
CONFIG_STACKSIZE="(128 * 1024)"
CONFIG_BOOTDELAY=2
CONFIG_SYS_HELP_CMD_WIDTH=8
CONFIG_SYS_TIMER_CTRL="0x84"
...

这些以CONFIG_开头的宏就是一系列的配置开关，会影响后继的编译过程。

根据Makefile文件下面的注释，autoconfig.mk文件会被所有的Makefile文件包含，具体说是因为所有子目录的Makefile都有这么一句话include (TOPDIR)/config.mk，即包含了顶层目录下的config.mk文件，而顶层目录下的config.mk文件里有这么一行sinclude $(OBJTREE)/include/autoconf.mk。。

autoconfig.mk.dep文件的内容是autoconfig.mk文件产生时所依赖的文件的列表，这个列表中的任意一个头文件发生修改变化都会导致automake.mk文件重新生成一次，其内容如下所示：

include/autoconf.mk: include/common.h \
 /home/fs/temp/u-boot-2009.11/include/config.h \
 /home/fs/temp/u-boot-2009.11/include/configs/versatile.h \
 /home/fs/temp/u-boot-2009.11/include/asm/config.h \
 /home/fs/temp/u-boot-2009.11/include/linux/bitops.h \
 /home/fs/temp/u-boot-2009.11/include/asm/types.h \

至于autoconfig.mk和autoconfig.mk.dep里面的内容是哪里来的，让我们来仔细看一下下面这两条规则：

#
# Auto-generate the autoconf.mk file (which is included by all makefiles)
#
# This target actually generates 2 files; autoconf.mk and autoconf.mk.dep.
# the dep file is only include in this top level makefile to determine when
# to regenerate the autoconf.mk file.
$(obj)include/autoconf.mk.dep: $(obj)include/config.h include/common.h
    @$(XECHO) Generating $@ ; \
    set -e ; \
    : Generate the dependancies ; \
    $(CC) -x c -DDO_DEPS_ONLY -M $(CFLAGS) $(CPPFLAGS) \
        -MQ $(obj)include/autoconf.mk include/common.h > $@

$(obj)include/autoconf.mk: $(obj)include/config.h
    @$(XECHO) Generating $@ ; \
    set -e ; \
    : Extract the config macros ; \
    $(CPP) $(CFLAGS) -DDO_DEPS_ONLY -dM include/common.h | \
        sed -n -f tools/scripts/define2mk.sed > $@.tmp && \
    mv $@.tmp $@

在自动make过程中，这两条规则会先执行第二条规则，运行预处理命令，选项”-dM”的作用是提取include/common.h中定义的宏，然后输出给tools/scripts/define2mk.sed脚本处理，提取所有的config marcos，即配置宏，保存到autoconf.mk文件中。

至于autoconf.mk的内容，因为include/common.h该文件又包含了$(obj)include/config.h，$(obj)include/config.h这个文件是我们执行make xxx_config生成的。以我操作的versatile开发板为例，执行make versatile_config后生成的’include/config.h’里面的内容如下：

#define CONFIG_ARCH_VERSATILE_PB

/* Automatically generated - do not edit */
#include <configs/versatile.h>
#include <asm/config.h>

所以autoconf.mk文件中出现的所有”CONFIG_”宏的来源就是来自于include/common.h，include/config.h，include/configs/versatile.h以及include/asm-arm/config.h等等一系列的头文件，由于头文件会嵌套包含，所以autoconf.mk里的这些宏究竟依赖与多少头文件，这就是列在autoconfig.mk.dep里的内容了。

当make xxxx_config的时候，譬如make versatilepb_conifg，下面的为目标%_config会被执行，先执行unconfig删除上次的配置文件，然后@$(MKCONFIG) -A $(@:_config=)，第一个@作用是抑制回显，$(MKCONFIG)就是执行mkconfig脚本。具体执行该脚本发生了什么，参考理解u-boot的make xxx_config机制和mkconfig(基于v2013.01)

%_config::    unconfig
    @$(MKCONFIG) -A $(@:_config=)

在2013.01版本里，相较于2009.11，这里做了较大的改进，本质思想就是把所有的xxxx_config规则定义在顶层目录下的boards.cfg文件里，然后在这里用awk抽取信息后展开而已。

awk处理中，(NF && $$1 !~ /^#/)，NF表示字段总数不为零，”&& $$1 !~ /^#/“表示而且第一个字段不是’#’开头的，总的意思是匹配不以’#’开头的行。{ print $$1 “: “ $$1 “_config; $$(MAKE)” }’，意思就是打印”$1 : $1_config; $$(MAKE)”, $< > $@意思就是把打印的字符串信息写入到目标$(obj).boards.depend中。
(补充知识 $@:表示规则的目标文件名 $<:表示规则的第一个依赖文件名 ‘>’为输出导向符)

sinclude $(obj).boards.depend
$(obj).boards.depend:    boards.cfg
    @awk '(NF && $$1 !~ /^#/) { print $$1 ": " $$1 "_config; $$(MAKE)" }' $< > $@
