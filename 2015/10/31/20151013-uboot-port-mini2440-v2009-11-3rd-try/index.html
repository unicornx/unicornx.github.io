针对mini2440开发板的U-boot移植第三次尝试笔记(v2009.11)

发表于 2015-10-31   |   分类于 Bootloader   |  

文章系原创，如需转载请注明：转载自”Blog of UnicornX” (http://unicornx.github.io/)

经过前面两轮实验后，本次作为基于mini2440，基于U-boot v2009.11的最后一次总结。
关于我的第一次尝试，请参考针对mini2440开发板的U-boot移植笔记(v2009.11)。
关于我的第二次尝试，请参考针对mini2440开发板的U-boot移植第二次尝试笔记(v2009.11)。

总体目标

首先确定一下我本次移植实验的目标：

  • 在第三次尝试中我将按照第一次尝试的顺序做，但加入第二次尝试的新的体会。
  • 将此前遗留的一些我觉得tekkaman做的不够好的地方适当按照自己的想法调整一下。
  • 只针对mini2440的SOC芯片S3C2440，不兼容S3C2410等，原来tekkaman要考虑兼容，所以代码里有很多的宏
  • 功能点实现上仍然和第一次尝试的保持一致。
    • 支持串口显示
    • 支持nandflash启动，支持nandflash的一些基本操作维护命令
    • 支持usb传输并在nandflash上烧写u-boot/kernel/rootfs,（注：对rootfs的大小目前有限制，不可以超过STACK_BASE/0x33F00000-PHYS_SDRAM_1/0x30000000 = 0x3F00000 = 63MB
    • 支持通过U-boot引导nandflash上的内核
    • 对norflash启动的支持，暂时不打算做了。

涉及的知识点：

  • execption/interrupt
  • clock 参考基于mini2440开发板学习S3C2440的系统时钟
  • gpio/led
  • sdram 参考基于mini2440开发板学习S3C2440的内存控制器
  • uart
  • nand-flash 参考基于mini2440开发板学习Nandflash
  • usb（涉及DMA）参考基于mini2440开发板学习DMA

大致的计划步骤如下：

步骤 目标
Step1 建立基本框架，为后继工作打下基础，仅保证编译通过
Step2 确保第一阶段和第二阶段流程基本正常。通过Norflash启动后利用supervivi对SDRAM初始化，然后通过supervivi将程序下载到sdram上运行，通过点灯确认流程的正确性。
Step3 在第一阶段，使能代码重定向，实现从nandflash启动，通过点灯确认采用代码重定向后第一阶段和第二阶段流程基本正常。
Step4 确保第二阶段的基本流程正确，通过点灯验证start_armboot基本流程正确，添加UART打印功能，为后面更复杂的功能调试做准备
Step5 为第二阶段添加nandflash功能
Step6 为第二阶段添加usb传输功能
Step7 在step5和step6的基础上从nandflash引导内核

Step2 确保总体流程运行正常

流程分析

先总结一下mini2440的启动部分的代码大致逻辑如下，总体上分为两个阶段：

@cpu/arm920t/start.S
_start:    b start_code
... //异常向量表
//第一阶段
start_code: 
{
    1.1> 针对第一阶段的一系列初始化动作，包括
    - 让系统进入SVC（管理员模式）
    - 关闭看门狗
    - 关闭所有中断
    - 设置时钟的分频比
    - cpu_init_crit
    {
        - 关闭MMU和cache    
        - lowlevel_init@board/unicornx/mini2440/lowlevel_init.S，初始化SDRAM
    }
    1.2> 代码重定向，如果不是在内存中(譬如从NOR-Flash或者Nandflash里，针对我这里移植的情况，指的就是从Nandflash启动，就将自己从Nand-Flash拷贝到SDRAM中
    1.3> 堆栈初始化，为第二阶段的C语言做准备
    1.4> 对BSS段清零，为第二阶段的C语言做准备
    1.5> 跳向第二阶段的C语言代码入口_start_armboot，1.2已经确保相关指令被重定向到内存
}

@lib_arm/board.c
//第二阶段
void start_armboot(void)
{
    ...
}

该Step的目标需要至少确认如下两点可以到达：
1) 板子启动后是否到达第一阶段的入口处- start_code
2) 确认流程是否跳转到第二阶段入口处 - start_armboot。

为简化该阶段的工作，尽量不涉及过多外设的操作。

  • 因为是直接下载到指定SDRAM位置运行，和nandflash无关，所以不需要使能nandflash，同时可以屏蔽第一阶段中的代码重定向。SDRAM的初始化我们依赖supervivi来操作所以也不需要初始化SDRAM。
  • 对第二阶段屏蔽主要执行逻辑，避免程序异常
  • 仅使用点灯的方式确认程序运行状态，在第一阶段结束处点LED1灯确认流程至此；在start_armboot函数的入口点LED2灯确认流程至此。此时还是单线程状态，比较好确认，主要关注异常跳转(第一阶段进入)和函数跳转(第二阶段进入)
  • 不涉及串口终端输出，即不涉及UART

该过程中应该会涉及

  • 中断处理，仅仅是初始化为关闭
  • 时钟的初始化设置，必须的。
  • GPIO/LED点灯，用于调试

测试

Norflash+supervivi启动后会对SDRAM初始化，然后通过supervivi和DWN将我的处于开发阶段的uboot下载到sdram上运行，supervivi菜单选d，DWN中的下载地址设置为TEXT_BASE，即0x33f80000。这时应该可以看到LED1和LED2亮了，说明程序正确运行了第一阶段并到达到第二阶段的入口start_armboot。下面

有关supervivi的d命令，参考”Mini2440 之Linux 移植开发实战指南(2010-4-14)”的”2.1.4 supervivi 功能一览表”章节可知功能[d]：通过USB 下载程序到指定内存地址(通过DNW 的Configuration->Option 选项指定运行地址)，并运行。对于本开发板，SDRAM 的物理起始地址是0x30000000，结束地址是0x34000000，大小为64Mbytes，另外BIOS 本身占用了0x33DE8000以上的空间，因此在用BIOS的USB下载功能时应指定地址在0x30000000 - 0x33DE8000 之间。。注：这里的BIOS应该指的就是supervivi。考虑到我们运行U-boot的时候指定的运行地址是0x33f80000，在supervivi的BIOS运行地址0x33DE8000的高地址方向，间隔有1632K，应该不会影响到USB下载期间的supervivi。

Step3 使能代码重定向

S3C2440的nandflash启动的流程是一旦上电后其内部的存储控制单元会自动从nandflash上拷贝前4K字节的指令到S3C2440内部的一块SRAM，称之为stepstone，这块SRAM映射到起始地址为0处并开始运行。我们的目标就是要实现从nandflash启动U-boot同时摆脱4K大小的限制。所以需要在前4K指令中实现所谓的代码重定向的功能：

  • 自动检测出当前指令运行在SRAM还是SDRAM
  • 初始化SDRAM
  • 读nandflash，写SDRAM，实现将U-boot的指令从nandflash拷贝到SDRAM的指定位置TEXT_BASE，即0x33f80000处。这个流程其实就和我们在Step2中supervivi的d指令有着异曲同工的作用。

整个过程目前仍然是通过点灯确认采用代码重定向后第一阶段和第二阶段流程基本正常。当然这里仍然要依赖supervivi将U-boot烧写到nandflash的指定区域，然后再测试。

void start_armboot (void)
{

    /* simply light on LED1 */
    struct s3c24x0_gpio * const gpio = s3c24x0_get_base_gpio();
    gpio->GPBCON = 1 << 10;
    gpio->GPBUP = 1 << 5;
    gpio->GPBDAT &= ~(1 << 5);
#endif /* CONFIG_MINI2440_LED */

    /* 初始化全局数据块gd_t以及bd_t */    
    init_fnc_t **init_fnc_ptr;
    char *s;
#if defined(CONFIG_VFD) || defined(CONFIG_LCD)
    unsigned long addr;
#endif

    gd = (gd_t*)(_armboot_start - CONFIG_SYS_MALLOC_LEN - sizeof(gd_t));

    memset ((void*)gd, 0, sizeof (gd_t));

整个start_armboot()函数可以分为两大块，
第一块是系统和各种外设的初始化，当前第二阶段对硬件的一次完整的初始化过程，总结如下：

  • 时钟 @board_init()
  • GPIO @board_init()
  • 中断 @interrupt_init()
  • timer @timer_init()
  • uart @serial_init()
  • flash @flash_init()，注，这里的flash指的是norflash。

注：第二阶段没有对SDRAM进行初始化，该工作已经由第一阶段的lowlevel_init完成。也许U-boot把RAM认为是相对于这些外设更lowlevel的设备提前进行了初始化。

第二块就是最后的一个大的循环。

/* main_loop() can return to retry autoboot, if so just run it again. */
for (;;) {
    main_loop ();
}

在这个循环中调用main_loop()在/common/main.c文件中定义。在main_loop函数中，有两条比较关键的路径需要分析，一条路径就是引导内核启动，另外一条路径就是用户通过串口输入中断了u-boot启动内核过程而进入了u-boot的命令行调试运行环境中。具体的代码比较长，而且充斥了宏开关，比较难看，但整理一下后大致的运行过程以类似C代码的方式总结如下：

void main_loop (void)
{
    static char lastcommand[CONFIG_SYS_CBSIZE] = { 0, };
    char *s;

    /* get boot delay seconds */
    s = getenv ("bootdelay"); 
    bootdelay = s ? (int)simple_strtol(s, NULL, 10) : CONFIG_BOOTDELAY;

    s = getenv ("bootcmd");

    /* 
     * 调用abortboot函数检查在等待的bootdelay秒内是不是
     * 从串口有用户输入了任意字符，如果有就说明用户想进入命
     * 令行调试模式，如果没有就说明可以通过调用run_command()
     * 开始运行启动内核的命令，具体的命令s通过bootcmd环
     * 境变量设置。
     * 其中abortboot()也是在`/common/main.c`文件中定义
     * 其主要逻辑是：在一个循环中不断检测是不是有用户的
     * 串口输入，使用了一个udelay函数，10000us等于10ms，
     * 循环100次就是1s
     */
    if (bootdelay >= 0 && s && !abortboot (bootdelay)) {

        run_command (s, 0);
    }

    /* 跳过自动引导后，进入命令行操作模式 */
    for (;;) {
        len = readline (CONFIG_SYS_PROMPT);

        flag = 0;    /* assume no special flags for now */
        if (len > 0)
            strcpy (lastcommand, console_buffer);
        else if (len == 0)
            flag |= CMD_FLAG_REPEAT;

        if (len == -1)
            puts ("<INTERRUPT>\n");
        else
            rc = run_command (lastcommand, flag);

        if (rc <= 0) {
            /* invalid command or not repeatable, forget it */
            lastcommand[0] = 0;
        }
    }
}

这里顺便记录一下有关如何关闭tekkaman打开的蜂鸣器，采用tekkaman制作的U-boot启动时总是会叫一声，如果想要关闭这个叫声，首先看看tekkaman的做法：程序的流程：查看board/unicornx/mini2440/mini2440.c文件中的init_sequence[]数组可知:

start_armboot()
    |_board_init() //这里tekkaman会打开一次蜂鸣器
    |_display_banner() //这里tekkaman会再打开一次蜂鸣器
    ......
    在所有外设初始化完成，进入命令行处理之前，tekkaman会关闭蜂鸣器

这就是我们在U-boot启动阶段会听到蜂鸣器响一下又灭掉的原因。如果要彻底不让蜂鸣器鸣叫，需要把board_init()和display_banner()两处的蜂鸣器都关闭。当然在我的第三次U-boot移植中因为已经不是照抄tekkaman的代码了，一开始就没有开蜂鸣器，所以无需考虑这个问题。这里只是顺便记录一下，备忘。
