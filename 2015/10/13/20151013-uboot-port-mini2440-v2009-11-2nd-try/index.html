针对mini2440开发板的U-boot移植第二次尝试笔记(v2009.11)

发表于 2015-10-13   |   分类于 Bootloader   |  

文章系原创，如需转载请注明：转载自”Blog of UnicornX” (http://unicornx.github.io/)

首先声明一下，其实第二次尝试的结果是失败的，虽然有解决的可能，但我最终还是放弃了。但整个过程却是给我积累了不少的经验，我想一定会为我的第三次尝试积累很好的经验。所以我还是将其记录下来。

关于我的第一次尝试，请参考针对mini2440开发板的U-boot移植笔记(v2009.11)。

总体目标

首先确定一下我的新的目标：

  • 在第二次尝试中我要继续第一次尝试中在“再出发”章节所构思的新的移植步骤。
  • 按照自己的思路把一些代码整理一下，不再严格和tekkaman的代码对齐了。但总的参考蓝本还是以友善之臂提供的Mini2440之U-boot使用及移植详细手册2010-4-16为主要蓝本，以tekkaman的2009.11的U-boot移植代码为主要代码参考。
  • 只针对mini2440的SOC芯片S3C2440，不兼容S3C2410等，原来tekkaman要考虑兼容，所以代码里有很多的宏
  • 功能点实现上和第一次尝试的保持一致。
    • 支持串口显示
    • 支持nandflash启动，支持nandflash的一些基本操作维护命令
    • 支持usb传输并在nandflash上烧写u-boot/kernel/rootfs,（注：对rootfs的大小目前有限制，不可以超过STACK_BASE/0x33F00000-PHYS_SDRAM_1/0x30000000 = 0x3F00000 = 63MB
    • 支持通过U-boot引导nandflash上的内核
    • (可选)支持norflash启动，支持norflash的一些基本操作维护命令，目前暂未实现。

涉及的知识点：

  • execption/interrupt
  • clock
  • gpio/led
  • sdram
  • uart
  • nand-flash
  • usb（涉及DMA）
  • nor-flash （可选）

大致的计划步骤如下：

结合mini2440提供的supervivi调试移植U-boot设计一个更好的流程。先移植调试第二阶段。即采用norflash+supervivi启动，然后将调试的U-boot直接下载到SDRAM中运行，调试直到可以引导内核。最后再调试第一阶段的搬运逻辑。

调整后移植的步骤如下：

步骤 目标
Step1 建立基本框架，为后继工作打下基础，仅保证编译通过
Step2 通过Norflash启动后利用supervivi对SDRAM初始化，然后通过supervivi将程序下载到sdram上运行，屏蔽第一阶段，通过点灯确认可以跳转到第二阶段的入口处。
Step3 通过点灯验证start_armboot基本初始化流程正确，添加UART打印功能，为后面更复杂的功能调试做准备
Step4 为第二阶段添加nandflash功能
Step5 为第二阶段添加usb传输功能。注：本次尝试最终止步于此。具体原因参考Step5的详细记录。
Step6 在step5和step6的基础上实现从nandflash引导内核
Step7 使能代码重定向，利用重定向将代码从nandflash搬移到SDRAM自动运行第二阶段，至此完成。

Step2 确认可以跳转到第二阶段

先总结一下mini2440的启动部分的代码大致逻辑如下，总体上分为两个阶段：

@cpu/arm920t/start.S
_start:    b start_code
... //异常向量表
//第一阶段
start_code: 
{
    1.1> 针对第一阶段的一系列初始化动作，包括
    - 让系统进入SVC（管理员模式）
    - 关闭看门狗
    - 关闭所有中断
    - 设置时钟的分频比
    - cpu_init_crit
    {
        - 关闭MMU和cache    
        - lowlevel_init@board/tekkamanninja/mini2440，初始化SDRAM
    }
    1.2> 代码重定向，如果不是在内存中(一般是在NOR-Flash里，就将自己从NOR-Flash拷贝到SDRAM中
    1.3> 堆栈初始化，为第二阶段的C语言做准备
    1.4> 对BSS段清零，为第二阶段的C语言做准备
    1.5> 跳向第二阶段的C语言代码入口_start_armboot，1.2已经确保相关指令被重定向到内存
}

@lib_arm/board.c
//第二阶段
void start_armboot(void)
{
    ...
}

由于该Step的目标仅仅是要确认跳转到第二阶段入口处即可，为简化该阶段的工作，尽量不涉及过多外设的操作。

  • 可以屏蔽第一阶段中的代码重定向，所以不需要使能nandflash和SDRAM，SDRAM的初始化我们依赖supervivi来操作。
  • 屏蔽第二阶段的执行逻辑，避免程序异常
  • 仅使用点灯的方式确认程序运行状态，所以不涉及串口终端输出，即不涉及UART

该过程中应该会涉及

  • 中断处理，仅仅是初始化为关闭
  • 时钟的初始化设置，必须的。
  • GPIO/LED点灯，用于调试

测试

Norflash+supervivi启动后会对SDRAM初始化，然后通过supervivi和DWN将我的处于开发阶段的uboot下载到sdram上运行，supervivi菜单选d，DWN中的下载地址设置为TEXT_BASE，即0x33f80000。这时应该可以看到LED1亮了，说明程序正确运行到第二阶段的入口start_armboot。接下来我们可以继续采用在SDRAM上调试第二阶段中的详细步骤，这样就用不着每次都烧写nandflash了，不仅方便，而且可以延长nandflash的读写寿命。

有关supervivi的d命令，参考”Mini2440 之Linux 移植开发实战指南(2010-4-14)”的”2.1.4 supervivi 功能一览表”章节可知功能[d]：通过USB 下载程序到指定内存地址(通过DNW 的Configuration->Option 选项指定运行地址)，并运行。对于本开发板，SDRAM 的物理起始地址是0x30000000，结束地址是0x34000000，大小为64Mbytes，另外BIOS 本身占用了0x33DE8000以上的空间，因此在用BIOS的USB下载功能时应指定地址在0x30000000 - 0x33DE8000 之间。。注：这里的BIOS应该指的就是supervivi。考虑到我们运行U-boot的时候指定的运行地址是0x33f80000，在supervivi的BIOS运行地址0x33DE8000的高地址方向，间隔有1632K，应该不会影响到USB下载期间的supervivi。

确保第二阶段的基本流程正确

主要参考lib_arm/board.c文件中的start_armboot()函数及其相关逻辑，其中各个模块的初始化都是通过很多宏开关来控制的，所有的宏开关可以参考include/configs/mini2440.h，include/config_cmd_default.h（这个文件被mini2440.h所包含）
其实最快的检查开关的方法是查看include/autoconf.mk，运行make后会产生这个文件，列出所有宏的开关值。改变这些宏开关后，只要重新make clean一下，再make就好了。

void start_armboot (void)
{
#ifdef CONFIG_MINI2440_LED
    /* simply light on LED1 */
    struct s3c24x0_gpio * const gpio = s3c24x0_get_base_gpio();
    gpio->GPBCON = 1 << 10;
    gpio->GPBUP = 1 << 5;
    gpio->GPBDAT &= ~(1 << 5);
#endif /* CONFIG_MINI2440_LED */

    /* 初始化全局数据块gd_t以及bd_t */    
    init_fnc_t **init_fnc_ptr;
    char *s;
#if defined(CONFIG_VFD) || defined(CONFIG_LCD)
    unsigned long addr;
#endif
    /* Pointer is writable since we allocated a register for it */
    gd = (gd_t*)(_armboot_start - CONFIG_SYS_MALLOC_LEN - sizeof(gd_t));
    /* compiler optimization barrier needed for GCC >= 3.4 */
    __asm__ __volatile__("": : :"memory");

    memset ((void*)gd, 0, sizeof (gd_t));
    gd->bd = (bd_t*)((char*)gd - sizeof(bd_t));
    memset (gd->bd, 0, sizeof (bd_t));

    gd->flags |= GD_FLG_RELOC;

    monitor_flash_len = _bss_start - _armboot_start;

    /* 
     * 根据init_sequence全局函数数组顺序执行初始化 
     *init_fnc_t *init_sequence[] = {
     *#if defined(CONFIG_ARCH_CPU_INIT)
     *    arch_cpu_init,        /* basic arch cpu dependent setup */
     *                /*
     *                 * mini2440没有定义这个宏
     *                 */
     *#endif
     *    board_init,        /* basic board dependent setup */
     *                /*
     *                 * board/../mini2440.c
     *                 * 用于设置一些板级别的外设初始化
     *                 * 主要包括时钟和GPIO
     *                 */
     *#if defined(CONFIG_USE_IRQ)
     *    interrupt_init,        /* set up exceptions */
     *                /*
     *                 * lib_arm/interrupts.c
     *                 * 初始化中断
     *                 */
     *#endif
     *    timer_init,        /* initialize timer */
     *                /*
     *                 * cpu/arm920t/s3c24x0/timer.c
     *                 * 初始化定时器
     *                 */
     *    env_init,        /* initialize environment */
     *                /*
     *                 * common/env_flash.c
     *                 * 初始化环境变量
     *                 */
     *    init_baudrate,        /* initialze baudrate settings */
     *                /*
     *                 * lib_arm/board.c
     *                 * 初始化波特率，从环境变量baudrate里读取
     *                 */
     *    serial_init,        /* serial communications setup */
     *                /*
     *                 * drivers/serial/serial_s3c24x0.c
     *                 * UART初始化
     *                 */
     *    console_init_f,        /* stage 1 init of console */
     *                /*
     *                 * common/console.c
     *                 * 初始化控制台 
     *                 */
     *    display_banner,        /* say that we are here */
     *                /*
     *                 * lib_arm/board.c
     *                 * 显示U-boot的版本信息version_string
     *                 * 以及自己的一些欢迎flag
     *                 */
     *#if defined(CONFIG_DISPLAY_CPUINFO)
     *    print_cpuinfo,        /* display cpu info (and speed) */
     *                /*
     *                 * mini2440没有定义这个宏
     *                 */
     *#endif
     *#if defined(CONFIG_DISPLAY_BOARDINFO)
     *    checkboard,        /* display board info */
     *                /*
     *                 * mini2440没有定义这个宏
     *                 */            
     *#endif
     *#if defined(CONFIG_HARD_I2C) || defined(CONFIG_SOFT_I2C)
     *    init_func_i2c,
     *                /*
     *                 * lib_arm/board.c
     *                 * 和I2C有关，mini2440没有定义这个宏
     *                 */            
     *#endif
     *    dram_init,        /* configure available RAM banks */
     *                /*
     *                 * board/../mini2440.c
     *                 */
     *#if defined(CONFIG_CMD_PCI) || defined (CONFIG_PCI)
     *    arm_pci_init,        
     *                /*
     *                 * lib_arm/board.c
     *                 * 和PCI有关，mini2440没有定义这个宏
     *                 */
     *#endif
     *    display_dram_config,
     *                /*
     *                 * lib_arm/board.c
     *                 * 显示和DRAM相关的信息“DRAM：xxxx”
     *                 */
     *    NULL,
     *};
     */
    for (init_fnc_ptr = init_sequence; *init_fnc_ptr; ++init_fnc_ptr) {
        if ((*init_fnc_ptr)() != 0) {
            hang ();
        }
    }

    /* armboot_start is defined in the board-specific linker script */
    mem_malloc_init (_armboot_start - CONFIG_SYS_MALLOC_LEN,
            CONFIG_SYS_MALLOC_LEN);

    /*
     * 控制台上显示Flash的大小“Flash：xxxx”
     */
#ifndef CONFIG_SYS_NO_FLASH

    display_flash_config (flash_init ());
#endif /* CONFIG_SYS_NO_FLASH */

#ifdef CONFIG_VFD
#    ifndef PAGE_SIZE
#      define PAGE_SIZE 4096
#    endif
    /*
     * reserve memory for VFD display (always full pages)

    addr = (_bss_end + (PAGE_SIZE - 1)) & ~(PAGE_SIZE - 1);
    vfd_setmem (addr);
    gd->fb_base = addr;
#endif /* CONFIG_VFD */

     * LCD

        gd->fb_base = addr;
    }
#endif /* CONFIG_LCD */

    puts ("NAND:  ");
    nand_init();        /* go init the NAND */
#endif

#if defined(CONFIG_CMD_ONENAND)
    onenand_init();
#endif

#ifdef CONFIG_HAS_DATAFLASH
    AT91F_DataflashInit();
    dataflash_print_info();
#endif

    /* initialize environment */
    env_relocate ();

#ifdef CONFIG_VFD
    /* must do this after the framebuffer is allocated */
    drv_vfd_init();
#endif /* CONFIG_VFD */

#ifdef CONFIG_SERIAL_MULTI
    serial_initialize();
#endif

    /* IP Address */
    gd->bd->bi_ip_addr = getenv_IPaddr ("ipaddr");

    stdio_init ();    /* get the devices list going. */

    jumptable_init ();

#if defined(CONFIG_API)
    /* Initialize API */
    api_init ();
#endif

    console_init_r ();    /* fully init console as a device */

#if defined(CONFIG_ARCH_MISC_INIT)
    /* miscellaneous arch dependent initialisations */
    arch_misc_init ();
#endif
#if defined(CONFIG_MISC_INIT_R)
    /* miscellaneous platform dependent initialisations */
    misc_init_r ();
#endif

    /* enable exceptions */
    enable_interrupts ();

    /* Perform network card initialisation if necessary */
#ifdef CONFIG_DRIVER_TI_EMAC
    /* XXX: this needs to be moved to board init */

    if (getenv ("ethaddr")) {
        uchar enetaddr[6];

    }
#endif /* CONFIG_DRIVER_SMC91111 || CONFIG_DRIVER_LAN91C96 */

    /* Initialize from environment */
    if ((s = getenv ("loadaddr")) != NULL) {
        load_addr = simple_strtoul (s, NULL, 16);
    }
#if defined(CONFIG_CMD_NET)
    if ((s = getenv ("bootfile")) != NULL) {
        copy_filename (BootFile, s, sizeof (BootFile));
    }
#endif

    board_late_init ();
#endif

    puts ("MMC:   ");
    mmc_initialize (gd->bd);
#endif

#ifdef CONFIG_BITBANGMII
    bb_miiphy_init();
#endif
#if defined(CONFIG_CMD_NET)
#if defined(CONFIG_NET_MULTI)
    puts ("Net:   ");
#endif
    eth_initialize(gd->bd);
#if defined(CONFIG_RESET_PHY_R)
    debug ("Reset Ethernet PHY\n");
    reset_phy();
#endif
#endif

#ifdef CONFIG_MINI2440_LED
    gpio->GPBDAT &= ~(1 << 8); /* light LED4 on */
#endif /* CONFIG_MINI2440_LED */

    /* main_loop() can return to retry autoboot, if so just run it again. */
    for (;;) {
        main_loop ();
    }

    /* NOTREACHED - no way out of command loop except booting */
}

整个start_armboot()函数可以分为两大块，
第一块是系统和各种外设的初始化，当前第二阶段对硬件的一次完整的初始化过程，总结如下：

  • 时钟 @board_init()
  • GPIO @board_init()
  • 中断 @interrupt_init()
  • timer @timer_init()
  • uart @serial_init()
  • flash @flash_init()，注，这里的flash指的是norflash。

注：第二阶段没有对SDRAM进行初始化，该工作应该交给第一阶段完成。在这里，我们先调试第二阶段，是由于利用supervivi内部的初始化内存的机制，所以这里不初始化也可以工作。

第二块就是最后的一个大的循环。

/* main_loop() can return to retry autoboot, if so just run it again. */
for (;;) {
    main_loop ();
}

在这个循环中调用main_loop()在/common/main.c文件中定义。在main_loop函数中，有两条比较关键的路径需要分析，一条路径就是引导内核启动，另外一条路径就是用户通过串口输入中断了u-boot启动内核过程而进入了u-boot的命令行调试运行环境中。具体的代码比较长，而且充斥了宏开关，比较难看，但整理一下后大致的运行过程以类似C代码的方式总结如下：

void main_loop (void)
{
    static char lastcommand[CONFIG_SYS_CBSIZE] = { 0, };
    char *s;

    /* get boot delay seconds */
    s = getenv ("bootdelay"); 
    bootdelay = s ? (int)simple_strtol(s, NULL, 10) : CONFIG_BOOTDELAY;

    s = getenv ("bootcmd");

    /* 
     * 调用abortboot函数检查在等待的bootdelay秒内是不是
     * 从串口有用户输入了任意字符，如果有就说明用户想进入命
     * 令行调试模式，如果没有就说明可以通过调用run_command()
     * 开始运行启动内核的命令，具体的命令s通过bootcmd环
     * 境变量设置。
     * 其中abortboot()也是在`/common/main.c`文件中定义
     * 其主要逻辑是：在一个循环中不断检测是不是有用户的
     * 串口输入，使用了一个udelay函数，10000us等于10ms，
     * 循环100次就是1s
     */
    if (bootdelay >= 0 && s && !abortboot (bootdelay)) {

        run_command (s, 0);
    }

    /* 跳过自动引导后，进入命令行操作模式 */
    for (;;) {
        len = readline (CONFIG_SYS_PROMPT);

        flag = 0;    /* assume no special flags for now */
        if (len > 0)
            strcpy (lastcommand, console_buffer);
        else if (len == 0)
            flag |= CMD_FLAG_REPEAT;

        if (len == -1)
            puts ("<INTERRUPT>\n");
        else
            rc = run_command (lastcommand, flag);

        if (rc <= 0) {
            /* invalid command or not repeatable, forget it */
            lastcommand[0] = 0;
        }
    }
}

仍然通过点灯的方法检查流程执行是否正确，具体的点灯checkpoint如下：

  • 在刚刚进入start_armboot()的地方点亮LED1，这个在Step2中已经实现。
  • 完成board_init()后点亮LED2
  • 进入display_banner()时点亮LED3，在此之前基本的初始化完成
  • 在进入控制台cmd循环之前点亮LED4

Step5 为第二阶段添加usb传输功能

注：本次尝试最终止步于此。

目前的修改的测试结果发现进入“Hit any key to stop autoboot:”显示后就无动作，倒计时时间停止在3不减少。
当时第一感觉的判断是检查common/main.c里的abortboot函数里为何无法走到line257 printf("\b\b\b%2d ", bootdelay);。经过跟踪打印日志发现出问题的地方是在common/main.c里的abortboot函数里

while ((bootdelay > 0) && (!abort)) {
    ...
    udelay(10000);
}

这里udelay函数运行几次后就不正常，并结合另外一个发现，就是在lib_arm/board.c文件中注释掉line410usb_init_slave();后倒计时就可以正确运行了。所以估计是USB设备的初始化部分导致实时时钟运行有问题。所以需要看看usb_init_slave();里是不是对实时时钟的初始化有问题。
经过倒排除的方法，目前最新的发现是在usb_init_slave函数里注释掉gpioregs->GPCDAT |= (1<<5); /* enable USB Device, thisway.diy */控制台的显示就ok，但这解决不了问题，因为注释掉的这句是关键的打开usb设备功能。

后来还尝试过在出问题的地方common/main.c里的abortboot函数里，用强制延时代替udelay

while ((bootdelay > 0) && (!abort)) {
    ...
    //udelay(10000);
    volatile ulong l = 100000;
    while(l--);
}

虽然可以实现倒计时正确显示，但发现一旦使用usaslave命令则显示USB host is not connected yet.，所以最新的想法是，由于第二次尝试中的一个主要思路就是将nandflash启动重定向的移植步骤后移，前面的步骤都是通过supervivi，经supervivi的USB下载功能将待调试的uboot的image下载到RAM中运行的，是不是由于supervivi已经开启并运行了USB功能，如果u-boot在RAM中运行时又初始化USB设备，是不是需要将usb device controller复位一下，否则U-boot接管了supervivi后usb device为何会工作不正常呢。因为相同的U-boot的image，如果是从nandflash启动，就工作的好好的，而从nandflash启动是不使用supervivi的，我曾经计划再试试加上nandflash启动后用当前的代码测试一下usb是否ok，但后来也没有做，我估计很大的可能还是在这种调试状态下应该需要想办法重新复位一下usb device的usb接口部分。简单的尝试了在usb init中先调用udc_disconnect,但似乎不管用。网上有一篇文章USB概述及S3C2440 USB DEVICE,里面有usb_reset的逻辑想看看是否管用，后来想想还是算了，就到此为止吧。

还有一个问题是记得在上次移植中drivers/usb/slave/usbinit.c中注释如下：

//printf("usb_init_slave() done!\n");
udelay(100000);

但不记得为何要这么做了，似乎记得是如果不这么做usb功能就工作不正常，是在step6中引入的。
