一、NAND flash的存储单元组织结构

NAND-Flash上存放的数据的最小单位是bit，我们可以想象每个bit只能表示0或者1两种状态。每个bit保存在独立的存储单元(memory cell)中，一般来说，一个cell中只能存储一个bit。这些cell以8个或者16个为一组，连成bit-line，即所谓的byte(x8)/word(x16)，这就是所谓NAND-Flash芯片的位宽。多个bit-line组成一个page，多个page会再形成一个block，多个block最终构成一片NAND-Flash芯片。

对U-boot的平台定制机制的理解(基于v2013.01)

发表于 2015-07-20   |   分类于 Bootloader   |  

文章系原创，如需转载请注明：转载自”Blog of UnicornX” (http://unicornx.github.io/)

最新更新于：2015-11-3

有关单个的Make分析，请参考：
理解u-boot的Makefile(基于v2013.01)
理解u-boot的顶层config.mk(基于v2013.01)
理解u-boot的make xxx_config机制和mkconfig(基于v2013.01)

和当初学习v2009.01的时候一样，分析完uboot的Makefile，以及相关的重要文件后，最后总结一下。相比较于2009.11版本，总体框架上感觉变化不大。主要的改动集中在：

  • 源代码的目录树有较大的调整，新增了一个arch目录，把原来顶层目录下的众多lib_xxx目录和cpu目录都归并到arch目录下，2013.01版本的arch目录的层次感更强了，完全按照arch->cpu->soc做了组织。譬如：

    arch
|_arm //arch下第一级是按照顶级架构分arm,avr32,m68000,mips等
| |_cpu //每一种arch下的cpu目录里按cpu核分，譬如都是ARM公司出品的cpu核心有arm1136，arm920t等
| | |_arm1136
| | |_...
| | |_arm920t //每一种cpu核心下按照soc分，如都是采用arm920t核的soc有at91(AT91RM9200)，s3c24x0系列
| |   |_at91
| |   |_...
| |   |_s3c24x0
| |_dts //设备树
| |_include //每一种soc的头文件
| | |_asm
| |   |_arch_at91
| |   |_...
| |   |_arch_s3c24x0
| |_lib
|_avr32
|_m68k
|_mips
|_...

各个级别的配置文件的放置目录调整如下：

级别 对应的配置文件(举例)
ARCH arch/arm/config.mk
CPU arch/arm/cpu/arm926ejs/config.mk
SOC arch/arm/cpu/arm926ejs/at91/config.mk
BOARD board/atmel/at91sam9m10g45ek/config.mk
  • Makefile中针对各个开发板的config规则，全部分离抽取出来集中到board.cfg文件中统一配置，这样使得主Makefile看上去更精简。
  • 统一规定了查找lds链接脚本文件的顺序规则，参考理解u-boot的Makefile(基于v2013.01)的”搜索链接脚本文件”章节。
